# 道具系统 系统策划案

**文档版本**：v1.0  **系统ID**：item_system  **优先级**：P0  **复杂度**：M
**所属游戏**：围住Boss（Surround the Boss）
**对应脚本**：ItemSystem.ts  **模块类型**：Class
**配置文件**：`src/data/items.json`, `src/data/shop_config.json`

---

## 一、定位与目标

### 1.1 系统定位

道具系统是"围住Boss"中支撑玩家策略灵活性和付费转化的重要系统。在核心玩法"放置障碍物→Boss移动→胜负判定"的回合制循环中，道具为玩家提供了突破困境、修正失误、缩短学习曲线的柔性手段。道具不改变核心规则，但在关键时刻为玩家提供额外的容错空间和策略选择，避免因一步失误导致整局失败的挫败感。

道具系统同时承担社交裂变和广告变现的商业功能。玩家通过分享给好友或观看激励视频广告获取道具，形成"社交传播→道具获取→游戏体验提升→更多社交传播"的正向循环。

### 1.2 核心目标

1. **降低挫败感**：通过撤销和提示功能，帮助卡关玩家找到突破口，减少因反复失败导致的流失。
2. **策略深度**：额外步数道具让玩家有更多操作空间去尝试不同的围堵路径，而非严格受限于关卡步数上限。
3. **社交裂变**：通过"分享得道具"机制驱动玩家主动传播游戏，实现用户增长。
4. **广告变现**：通过"观看激励视频得道具"机制提供广告收入，同时玩家因主动选择观看广告而对广告接受度更高。
5. **游戏内经济闭环**：道具作为有限资源，其获取和使用形成轻量级的经济循环，为后续商业化扩展预留空间。

### 1.3 道具定位三原则

- **辅助性**：道具只能帮助玩家更容易地达成胜利，但不能替代核心玩法思考。"提示"道具给出推荐位置但不自动放置；"额外步数"道具增加步数但不改变围堵策略本身。
- **有限性**：每局每种道具的使用次数严格受限，确保道具经济不膨胀，同时保留付费和广告转化的上限空间。
- **时效性**：道具仅在当前对局中存在并获得意义。带入场内的道具消耗后即消失，不累积到下一关。道具获取（分享/广告）的次数按日刷新，驱动每日活跃。

### 1.4 与其他系统的关系

| 系统ID | 关系类型 | 具体说明 |
|--------|---------|---------|
| `gameplay_core` | 依赖 | 道具（额外步数、撤销、重置、提示）的作用目标为核心玩法的回合状态机、棋盘状态和步数计数器 |
| `main_menu` | 双向 | 主界面展示玩家当前持有道具数量；主界面分享成功后道具系统添加道具 |
| `level_result` | 输出 | 结算界面可展示本局使用的道具统计，并推荐玩家观看广告获取道具重试 |

| `ad_system` | 输入 | 玩家观看激励视频广告完成后回调道具系统，发放广告奖励道具 |
| `GameData` | 依赖 | 道具库存数据（`items`）、每日刷新数据（`dailyShareCount`, `dailyAdCount`）挂载在GameData单例上 |
| `analytics_system` | 输出 | 道具获取和使用行为上报埋点 |

## 三、核心玩法规则

### 3.1 道具类型定义

| 道具ID | 道具名 | 功能描述 | 每局使用上限 | 效果持续时间 | 触发时机 |
|--------|-------|---------|-------------|-------------|---------|
| `extra_step` | 多走1步 | 当前关卡步数上限增加5步，玩家可继续放置障碍物 | 2次 | 本局剩余所有回合 | 玩家回合（playerTurn），使用后步数上限即时增加 |
| `undo` | 返回上1步 | 恢复至上一步的完整棋盘状态（含障碍物布局、Boss坐标、步数计数），可连续使用回退多步 | 3次 | 单次回退操作 | 玩家回合（playerTurn），使用后立即生效 |
| `reset` | 重置本关 | 将当前关卡恢复到初始状态（障碍物布局、Boss起点、步数上限均恢复至开局配置），相当于重新开始本局 | 不限（取决于持有数量） | 替换当前对局 | 任意非结算状态，使用后重新加载关卡 |
| `hint` | 提示 | 基于当前棋盘状态和A*算法，计算一个最优的下一放置点，以高亮闪烁标记在棋盘上供玩家参考。不自动放置 | 1次 | 持续高亮直到玩家完成下一次放置，或玩家手动关闭 | 玩家回合（playerTurn），使用后棋盘上出现闪烁标记 |

### 3.2 道具获取规则

#### 3.2.1 全局持有

道具存在于玩家账户全局库存中（存储在 `GameData.items` 字典内），可以在主界面查看当前持有数量。道具进入对局后从全局库存中扣除，进入对局内临时计数。

#### 3.2.2 获取途径

| 途径 | 触发条件 | 奖励内容 | 每日上限 | 备注 |
|------|---------|---------|---------|------|
| 每日分享奖励 | 主界面点击分享并成功 | 随机获得一种道具 × 1 | 每日1次 | 每日UTC+8 0点重置 |
| 对局内分享 | 关卡中使用"+"获取 → 选择分享 → 分享成功 | 对应道具 × 1 | 每局限1次 | 不可与每日分享奖励叠加 |
| 激励视频广告 | 关卡中使用"+"获取 → 选择看广告 → 广告完成 | 对应道具 × 1 | 每日3次 | 每日UTC+8 0点重置 |
| 初始赠送 | 玩家首次进入游戏 | 每种道具各 × 1 | 仅1次 | 新用户 onboarding 奖励 |

#### 3.2.3 道具库存上限

| 道具类型 | 单种最大持有量 | 达到上限后行为 |
|---------|-------------|-------------|
| 多走1步 | 10 | 获取时弹出提示"该道具已达上限"，不增加数量 |
| 返回上1步 | 10 | 同上 |
| 重置本关 | 10 | 同上 |
| 提示 | 10 | 同上 |

### 3.3 道具使用规则

#### 3.3.1 使用流程

```
玩家点击道具图标
  ├─ 库存数量 > 0 且 本局已使用次数 < 每局上限
  │    └─ 执行道具效果 → 全局库存 -1 → 本局已使用次数 +1 → 更新UI
  └─ 库存数量 = 0 或 本局已使用次数达到上限
       └─ 展示Toast提示（具体文案见3.3.4）
```

#### 3.3.2 道具作用时机约束

| 道具 | 可用状态 | 不可用状态 |
|------|---------|-----------|
| `extra_step` | playerTurn（玩家回合） | bossMoving, judging, victory, defeatEscaped, defeatOutOfSteps, paused |
| `undo` | playerTurn，且 `undoHistory` 中存在至少一条记录（非第一步） | 无历史记录时按钮置灰 |
| `reset` | 任何非结算状态（loading除外） | loading, victory, defeatEscaped, defeatOutOfSteps（结算已出时无需重置） |
| `hint` | playerTurn，且棋盘仍有空位 | bossMoving, judging, 棋盘已无空位 |

#### 3.3.3 道具效果详细逻辑

**extra_step（多走1步）**：
- 效果：当前关卡允许的最大步数 `maxSteps` 增加 5。
- 步数计数器 `stepsLeft` 同步增加 5（即 `stepsLeft = stepsLeft + 5`）。
- 叠加规则：每局最多使用 2 次，即最多额外获得 10 步。
- UI反馈：步数指示器闪烁 +5 绿色数字动画（500ms），步数上限显示即时更新（如 "18/35" → "23/40"）。

**undo（返回上1步）**：
- 效果：从 `undoHistory` 栈中弹出最近一条棋盘快照，恢复至该快照状态。
- 恢复内容：障碍物布局（含预置障碍物和玩家放置的障碍物）、Boss坐标、`stepsLeft` 步数、`stepIndex`。
- 连续使用：可连续点击多次 undo 依次回退多步，直到 `undoHistory` 为空（回到第一步之前的状态）。
- 边界保护：当玩家正处于第一步（`stepIndex == 0` 或 `undoHistory` 为空）时，按钮置灰不可点击。
- 数据一致性：撤销后 `stepIndex` 回退 1，`stepsLeft` 恢复至撤销前的值（即归还本步消耗的步数）。
- UI反馈：棋盘从当前状态过渡到恢复状态的动画（200ms 淡入过渡），Boss位置从当前位置瞬移至恢复位置（或播放一段反向移动路径动画，P1优化）。

**reset（重置本关）**：
- 效果：重新加载当前关卡配置，棋盘、Boss、步数完全恢复至初始状态。
- 实现方式：调用 `LevelManager.loadLevel(currentLevelId)` 重新初始化。
- `undoHistory` 清空。
- 步数上限恢复至关卡配置的初始值（含已使用的额外步数道具效果被清除）。
- 道具库存中"重置本关"的数量 -1。
- UI反馈：棋盘淡出（300ms）→ 重置加载 → 棋盘淡入（300ms），过渡自然不突兀。

**hint（提示）**：
- 效果：基于当前棋盘状态和A*算法，计算一个最优的下一障碍物放置点。该位置是A*在模拟放置后使得Boss逃脱难度最大的交点。
- 算法逻辑：遍历所有空闲交点，模拟放置障碍物后运行A*，选择使Boss寻路路径最长或出口数量最少的位置。若存在多个同等优先级的位置，选择距离Boss最近的那个。
- 表现形式：推荐位置以脉冲呼吸灯动画高亮标识（周期 800ms，透明度 0.5→1.0→0.5 循环）。
- 交互：玩家可无视提示自行放置其他位置，也可点击提示位置放置。玩家完成一次放置后提示标记自动消失。
- 若玩家使用提示后暂时不操作，提示持续显示直到下一次放置或玩家切换到其他界面。
- 冷却：每局仅可使用1次提示，防止过度依赖。

#### 3.3.4 三态UI规则

道具图标在关卡界面底部道具栏中展示，根据状态呈现三种视觉形态：

| 状态 | 条件 | 视觉表现 | 交互行为 |
|------|------|---------|---------|
| `available` | 库存数量 > 0 且 本局已使用次数 < 每局上限 | 图标全彩显示，左下角显示当前全局库存数量（白色数字，红色背景圆点） | 点击图标 → 执行道具功能 |
| `unavailable` | 库存数量 = 0 且 本局已使用次数 < 每局上限 | 图标彩色但右上角显示"+"标记（白色加号，绿色圆底），左下角无数量 | 点击"+" → 弹出道具获取弹窗（IT-M5） |
| `exhausted` | 本局已使用次数 >= 每局上限 | 图标置灰（转为灰度图），左下角显示"已满"小字或 Max 标记，右下角斜杠覆盖 | 点击 → 弹出Toast提示"本局已达使用上限"（持续 2 秒后自动消失） |

三态优先级：`exhausted` > `unavailable` > `available`（即如果库存为0且已达上限，显示为 exhausted 态）。

### 3.4 每日重置规则

- 重置以 UTC+8 时区为准，每日 00:00:00 执行。
- 重置内容：
  - `dailyShareCount` 归零（主界面分享次数重置）。
  - `dailyAdCount` 归零（激励视频广告次数重置）。
- 重置判定方式：每次需要读取 `dailyShareCount` 或 `dailyAdCount` 时，比较记录中的 `lastResetDate` 与当前 UTC+8 日期：
  - 若 `lastResetDate < today` → 执行重置逻辑，将 `lastResetDate` 更新为 `today`。
  - 若 `lastResetDate == today` → 不做任何操作。
- 重置不由定时器触发，而是在每次道具系统初始化或写入时惰性检查（lazy check），避免不必要的后台计算。
- `lastResetDate` 存储格式：ISO日期字符串 `"2026-05-09"`。

### 3.5 道具生命周期（单局视角）

```
[对局前]          [对局中]                              [对局结束]
   │                 │                                      │
   ▼                 ▼                                      ▼
全局库存        道具进入对局                             对局内道具状态清空
(items)    ──►  从全局库存中扣除                   ──►  （仅持久化全局库存变动）
                 存入对局内临时计数                       未用完的道具不退回全局
                 对局中使用的道具实时扣减
                 对局中获取的道具即时加入（边玩边得）
```

关键规则：
- **带入与消耗**：进入关卡时从全局库存扣除的道具，如果在对局中未被使用，**不退还**到全局库存。这是经济设计的核心——鼓励玩家在合适的时机使用道具，而非囤积。
- **对局内获取**：在对局中通过分享或广告获取的道具，即时计入对局内可用次数（不经过全局库存）。该获取不影响"已带入"的道具数量，是对局中的额外增益。

## 四、完整流程与状态机

### 4.1 道具系统内部状态机

```
                     ┌──────────────────────┐
                     │       IDLE           │ ← 等待道具操作指令
                     └──────┬───────────────┘
                            │ 玩家点击道具图标（available态）
                            v
                     ┌──────────────────────┐
                     │   EXECUTING          │ ← 执行道具效果逻辑
                     └──────┬───────────────┘
                            │
              ┌─────────────┼──────────────┐
              │             │              │
              v             v              v
     ┌──────────────┐ ┌──────────┐ ┌──────────────┐
     │ EXTRA_STEP   │ │  UNDO    │ │   HINT       │
     │ (步数增加中)  │ │ (回退中)  │ │ (提示计算中)  │
     └──────┬───────┘ └────┬─────┘ └──────┬───────┘
            │              │              │
            └──────────────┴──────────────┘
                           │
                           v
                    ┌──────────────────────┐
                    │  UI_UPDATE           │ ← 更新UI状态、触发事件
                    └──────┬───────────────┘
                           │
                           v
                    ┌──────────────────────┐
                    │       IDLE           │ ← 回到等待态
                    └──────────────────────┘
```

**reset 道具特殊流程**：
```
IDLE → 点击 reset → 弹窗确认"确定要重置本关吗？进度将丢失"
  ├─ 玩家点击"确定" → 重置关卡 → 回到 IDLE
  └─ 玩家点击"取消" → 回到 IDLE（无任何操作）
```

### 4.2 道具获取弹窗流程（IT-M5）

```
玩家点击道具图标（unavailable态，即库存为0且未达上限）
  └─ [道具获取弹窗] 弹出，展示以下信息：
       ├─ 道具图标 + 道具名称
       ├─ 道具效果描述
       ├─ 获取方式按钮A："分享给好友"（调用 wx.shareAppMessage）
       │    └─ 分享成功后 → 发放道具 × 1 → 关闭弹窗 → 道具转为 available
       │         └─ 分享失败 → Toast"分享失败，请重试" → 弹窗不关闭
       ├─ 获取方式按钮B："观看视频广告"（调用 wx.createRewardedVideoAd）
       │    └─ 广告播放完成（onClose 且 isEnded）→ 发放道具 × 1 → 关闭弹窗
       │         └─ 广告未完成（提前关闭）→ Toast"看完广告才能获得奖励哦" → 弹窗不关闭
       └─ 关闭按钮（×）：手动关闭弹窗，不获取道具

发放道具逻辑：
  道具直接加入对局内可用次数（不经过全局库存）
  若该道具已有库存 > 0（罕见情况），则在原库存基础上 +1
```

### 4.3 详细业务流程：从点击到生效

**场景：玩家在 playerTurn 状态下使用"多走1步"道具**

1. 玩家点击道具栏中"多走1步"图标。
2. ItemSystem 检查条件：
   - `items['extra_step'] > 0`（全局库存）→ 通过。
   - `sessionItemUsage['extra_step'] < maxPerSession['extra_step']`（本局已用次数 < 2）→ 通过。
3. 系统状态转入 `EXECUTING`。
4. 执行效果：
   - `maxSteps += 5`，`stepsLeft += 5`。
   - 关卡界面顶部步数指示器更新显示。
5. 更新数据：
   - `items['extra_step'] -= 1`（全局库存减1）。
   - `sessionItemUsage['extra_step'] += 1`（本局已用次数加1）。
6. 触发事件 `itemUsed{ itemType: 'extra_step', quantityChange: -1, sessionUsage: 1 }`。
7. 状态回到 `IDLE`。
8. 若使用后库存变0：道具图标立即从 `available` 切换为 `unavailable`（显示"+"）。

**场景二：玩家使用"提示"道具**

1. 玩家点击"提示"道具图标（available 态）。
2. 条件检查通过后，系统调用 A* 模拟器遍历所有空闲交点。
3. 计算耗时 < 100ms（预期），玩家感知为即时反馈。
4. 推荐位置以呼吸灯动画在棋盘上标记。
5. 道具库存减1，本局使用次数加1。
6. 提示标记持续显示，直到玩家完成下一次障碍物放置。
7. 玩家放置后，标记自动消失，`hint` 状态解除。

### 4.4 边界流程

**场景：玩家使用 undo 回退到起始状态**

1. 玩家连续使用 undo 回退。
2. 当 `undoHistory` 中仅剩最后一条记录时，继续点击 undo。
3. 系统恢复至初始棋盘状态，`stepsLeft` 恢复为初始值，`stepIndex` 归零。
4. 此时 `undoHistory` 清空。
5. 道具图标（undo）从 `available` 变为 `exhausted`（若已达3次上限）或保持 `available`（若未达上限但库存有剩余）。
6. 若再次点击 undo 但 `undoHistory` 已空 → 弹出 Toast"没有可撤销的步骤"。

## 五、边界条件与异常处理

### 5.1 数值边界

| 编号 | 边界场景 | 预期行为 |
|------|---------|---------|
| B1 | 使用 extra_step 时 `maxSteps` 已达关卡预设上限 + 10（2次叠加上限） | 按钮置灰显示 exhausted 态，点击反馈"本局已达额外步数上限" |
| B2 | 使用 undo 时棋盘已处于初始状态（undoHistory 为空） | undo 按钮置灰（不可点击），不响应点击事件 |
| B3 | 使用 hint 时棋盘所有交点已被占据 | 提示"棋盘已满，无法计算提示位置"，按钮不消耗库存 |
| B4 | reset 确认弹窗中玩家点击"确定"时对局已结算 | 弹窗仍有效，重置后重新进入该关卡 |
| B5 | 同时拥有 extra_step 全局库存 > 0 且本局已达2次上限 | 显示 exhausted 态（本局上限优先于库存） |
| B6 | 道具库存 + 对局内获取累计达到单种上限 10 | 获取时提示已达上限，不再增加数量 |

### 5.2 异常处理

| 编号 | 异常类型 | 处理策略 |
|------|---------|---------|
| E1 | 使用 undo 时序列化快照数据损坏（JSON 解析失败） | Try/catch 捕获异常，弹出 Toast"数据异常，无法撤销"，保留当前棋盘状态不变，记录错误到 pipeline_errors.json |
| E2 | hint 计算时 A* 模拟器超时（> 200ms） | 中断计算，弹出 Toast"提示计算超时，请稍后再试"，不消耗道具库存 |
| E3 | 道具获取弹窗中选择"看广告"但广告加载失败 | 弹窗内显示"广告加载失败，请重试"，提供重试按钮，保留弹窗不关闭 |
| E4 | 道具获取弹窗中选择"分享"但微信分享 API 调用失败 | 弹窗内显示"分享失败，请重试"，保留弹窗，不发放道具 |
| E5 | localStorage 道具数据损坏或字段缺失 | 读取时捕获异常，使用 `items.json` 中的默认值兜底（各道具初始为0），静默修复 localStorage |
| E6 | 对局中道具库存出现负数（逻辑 bug） | 每次扣减操作前做 `Math.max(0, count - 1)` 防止下溢，同时输出 console.error 日志 |
| E7 | 每日重置时 lastResetDate 为空或格式异常 | 视为需要重置，将 lastResetDate 设为当前 UTC+8 日期，重置所有次数 |

### 5.3 幂等与容错

| 操作 | 幂等策略 |
|------|---------|
| 每日重置 | 基于日期字符串比较，同一日期多次调用不重复重置 |
| 道具发放 | 分享和广告回调中增加 `rewardClaimed` 标记（每个 rewardId 仅发放一次），防止回调多次导致重复发放 |
| 道具消耗 | 每次消耗前重新检查库存和本局次数，避免并发操作下的超扣 |

## 六、前后端通信设计

### 6.1 通信概览

道具系统为纯客户端系统，无服务端实时通信。所有数据存储在 `GameData` 单例中并通过 `localStorage` 持久化。分享和广告功能通过微信小游戏原生 API（`wx.shareAppMessage`, `wx.createRewardedVideoAd`）实现，道具系统以回调方式接收结果。

| 数据存储 | 键值 | 类型 | 初始值 | 说明 |
|---------|------|------|--------|------|
| 道具全局库存 | `GameData.items` | `Record<string, number>` | `{}` | 如 `{ extra_step: 2, undo: 1, reset: 0, hint: 3 }` |
| 每日分享次数 | `GameData.dailyShareCount` | `number` | `0` | 每日 UTC+8 0点重置 |
| 每日广告次数 | `GameData.dailyAdCount` | `number` | `0` | 每日 UTC+8 0点重置 |
| 每日重置日期 | `GameData.lastResetDate` | `string` | `""` | ISO 日期格式 `"2026-05-09"` |
| 本局道具用量 | `GameData.sessionItemUsage` | `Record<string, number>` | `{}` | 对局开始时初始化为空，结束时清空 |
| 撤销历史 | `GameData.undoHistory` | `BoardSnapshot[]` | `[]` | 每步放置前推入快照 |

### 6.2 事件总线协议

道具系统通过全局事件总线对外派发和监听以下事件，所有事件采用 camelCase 命名：

| 事件名 | 方向 | 载荷 | 触发时机 |
|--------|------|------|---------|
| `itemUsed` | ItemSystem → 全局 | `{ itemType: string, quantityChange: number, sessionUsage: number, roundId: string }` | 道具效果执行完成时 |
| `itemAcquired` | ItemSystem → 全局 | `{ itemType: string, quantity: number, source: 'share' \| 'ad' \| 'initial_gift', roundId?: string }` | 玩家成功获取道具时 |
| `itemLimitReached` | ItemSystem → UI | `{ itemType: string, reason: 'session_limit' \| 'inventory_full' }` | 道具已达使用上限或库存上限时 |
| `itemStateChanged` | ItemSystem → UI | `{ itemType: string, oldState: string, newState: string }` | 道具在三态间切换时 |
| `undoPerformed` | ItemSystem → UI/Core | `{ stepsRestored: number, restoredStepsLeft: number, boardSnapshot: BoardSnapshot }` | 撤销操作完成时 |
| `hintCalculated` | ItemSystem → UI | `{ suggestedPosition: Position, confidence: number, elapsedMs: number }` | 提示计算完成，返回推荐位置时 |
| `hintDismissed` | ItemSystem → UI | `{}` | 提示标记被清除（玩家放置后或手动关闭）|
| `dailyResetPerformed` | ItemSystem → 分析 | `{ date: string, resetShareCount: boolean, resetAdCount: boolean }` | 每日重置执行时 |
| `itemPopupOpened` | ItemSystem → 分析 | `{ itemType: string, source: 'in_game' \| 'main_menu' }` | 道具获取弹窗打开时 |
| `itemPopupClosed` | ItemSystem → 分析 | `{ itemType: string, action: 'share' \| 'ad' \| 'close' }` | 道具获取弹窗关闭时 |
| `itemAcquisitionFailed` | ItemSystem → 分析 | `{ itemType: string, source: 'share' \| 'ad', error: string }` | 道具获取失败时 |

## 九、迭代规划

### 9.1 P0（当前版本）

- IT-M1 道具库存管理：四种道具的全局存储、增减、查询、上限检查。
- IT-M2 对局道具管理：每局限次计数、进入对局时从全局扣除、对局结束时释放（不退还）。
- IT-M3 道具效果执行器：四种道具的核心效果逻辑（extra_step 加步数、undo 快照回退、reset 重载关卡、hint A*模拟计算推荐位置）。
- IT-M4 道具状态UI：关卡底部道具栏，完整三态（available / unavailable / exhausted）展示和交互。
- IT-M6 每日刷新调度器：惰性检查 UTC+8 日期变更，重置分享和广告次数。
- IT-M7 道具配置加载器：从 `items.json` 和 `shop_config.json` 读取配置。
- 初始赠送道具逻辑（新用户首次进入游戏时各道具 × 1）。

### 9.2 P1（核心体验完善）

- IT-M5 道具获取弹窗：完整 UI 实现，对接分享和广告回调。
- 对局内道具获取（玩家在关卡中点击"+"号弹窗获取道具）。
- 道具使用动画和特效完善（extra_step +5 动画、undo 棋盘闪烁、hint 呼吸灯等）。
- reset 确认弹窗，防止误触。
- 主界面道具展示区（显示四种道具当前持有数量）。
- 分享奖励随机选取（每日首次分享时从四种道具中随机选一种）。
- 每日分享红点提示（主界面分享按钮，首次进入且有分享次数时显示红点）。

### 9.3 P2（商业化和深度优化）

- 广告冷却机制：两次广告之间 30 秒冷却，防止玩家短时间内连续使用广告刷道具。
- 道具组合推荐：在失败结算界面根据对局数据分析推荐道具（"你只差 3 步就赢了，试试多走 1 步道具！"）。
- 道具限时活动：特定活动期间特定道具获取概率提升或每局上限临时增加。
- 道具背包扩展：付费扩展单种道具持有上限（10→20）。
- 批量获取优化：分享/广告成功后增加二次确认是否立即使用，或将其存入全局库存。
- 道具效果示意：使用前弹出道具效果预览（如使用 extra_step 前提示"使用后将增加 5 步剩余步数"）。
- 提示算法优化：引入机器学习轻量级模型（如规则引擎）替代简单的 A* 模拟，提升推荐质量（P2 探索性方向）。

### 9.4 明确不做

| 事项 | 不做原因 |
|------|---------|
| 道具合成/升级系统 | 与"围住Boss"的休闲定位不符，增加理解成本，偏离核心解谜体验 |
| 付费直接购买道具 | 游戏前期以裂变和广告为主，直接付费购买为时过早；保留为 P3 探索方向 |
| 道具交易/赠送好友 | 社交复杂度高且监管风险大（涉及虚拟财产），在微信小游戏环境中不适用 |
| 限时消失道具 | 道具应让玩家在需要时自由使用，加入时限会制造焦虑感，与休闲游戏调性不符 |
| 多语言道具描述 | MVP 阶段仅支持中文，多语言待后续版本根据海外发行计划决定 |
