# 分享系统 系统策划案

**文档版本**：v1.0  **系统ID**：share_system  **优先级**：P1  **复杂度**：S
**所属游戏**：围住Boss（Surround the Boss）
**对应脚本**：ShareSystem.ts  **模块类型**：Module
**依赖微信API**：`wx.shareAppMessage`, `wx.shareImageToMoments`, `wx.onShareAppMessage`

---

## 一、定位与目标

### 1.1 系统定位

分享系统是"围住Boss"社交裂变引擎的核心执行者，承担将玩家的游戏行为转化为社交传播事件的关键职责。在微信小游戏的生态中，分享是最自然、最高效的用户增长渠道——每个玩家的每一次分享，都是将游戏曝光推送到其微信好友关系链中的一次免费传播。分享系统不只是一个"转发工具"，而是一套围绕"分享动机设计→分享内容生成→分享渠道分发→分享奖励闭环"的完整社交传播引擎。

具体而言，分享系统需要解决以下四个核心问题：

1. **传播驱动**：将"围住Boss"的游戏过程（胜利、失败、日常进入）转化为有吸引力的社交内容，驱动玩家主动分享到微信好友和朋友圈。
2. **裂变增长**：通过分享卡片上的文案和图片激发接收者的好奇心和挑战欲，引导非玩家点击卡片进入游戏，实现用户获取的病毒式增长。
3. **奖励闭环**：通过"分享得道具"的即时奖励机制强化分享行为，形成"分享→收益→继续游玩→再次分享"的正向循环，同时通过每日限次控制奖励通胀。
4. **场景化适配**：针对主菜单、胜利结算、失败结算三种截然不同的情绪场景，生成匹配当下玩家心态和游戏状态的分享内容，最大化分享意愿和点击转化率。

### 1.2 核心目标

| 目标维度 | 具体指标 | 衡量方式 |
|---------|---------|---------|
| 分享渗透率 | 日分享用户数 / DAU >= 25% | 埋点 `shareTriggered` 去重用户数 |
| 分享回召率 | 通过分享卡片进入游戏的用户数 / 分享卡片曝光数 | 微信分享统计 |
| 分享奖励完成率 | 每日首次分享完成率（发起分享→成功回调）>= 85% | `shareCompleted` / `shareTriggered` |
| 主菜单分享转化率 | 主菜单"分享有礼"按钮点击后完成分享 >= 70% | 埋点追踪 |
| 分享再战转化率 | 失败界面点击"分享再战"后完成分享 >= 60% | `shareCompleted` + `retryRequested` |

### 1.3 与其他系统的关系

| 系统ID | 关系类型 | 具体说明 |
|--------|---------|---------|
| `main_menu` | 调用方 | 主菜单"分享有礼"按钮触发分享，分享成功后发放道具奖励 |
| `level_result` | 调用方 | 胜利界面"我要分享"和失败界面"分享再战"触发分享，分享行为关联不同后续逻辑 |
| `item_system` | 输出 | 主菜单分享成功后回调道具系统发放随机道具（每日首次） |
| `gameplay_core` | 输出 | 失败"分享再战"分享完成后触发关卡重开 |
| `GameData` | 读写 | 读取 `dailyShareCount` 判断每日限次，写入 `dailyShareCount` 递增 |
| `leaderboard` | 协同 | 排行榜"邀请好友"按钮复用分享系统的分享入口 |
| `微信小游戏API` | 基础设施 | `wx.shareAppMessage`（好友/群）、`wx.shareImageToMoments`（朋友圈） |

## 二、功能模块拆解

### 2.1 模块总览

| 模块ID | 模块名称 | 职责描述 | 优先级 | 依赖 |
|--------|---------|---------|--------|------|
| SH-M1 | 分享入口调度器 | 管理三个分享入口（主菜单/胜利/失败）的触发逻辑，根据来源决定分享配置和回调行为 | P1 | 无 |
| SH-M2 | 分享内容生成器 | 针对不同分享场景生成差异化的分享文案、图片和跳转路径；支持 Canvas 截图生成分享图 | P1 | Canvas渲染 |
| SH-M3 | 好友分享执行器 | 封装 `wx.shareAppMessage` API，将分享配置转发到微信好友/群聊 | P1 | 微信API |
| SH-M4 | 朋友圈分享执行器 | 封装 `wx.shareImageToMoments` API（或通过图片保存引导），将分享图片发布到朋友圈 | P2 | 微信API |
| SH-M5 | 分享奖励发放器 | 主菜单分享成功后判定每日限次，调用道具系统发放随机道具；管理"已分享"状态标记 | P1 | `item_system`, `GameData` |
| SH-M6 | 分享回调处理器 | 统一处理微信分享API的回调结果（成功/取消/失败），执行对应的后续动作（发奖/跳转/界面更新） | P1 | 无 |
| SH-M7 | 分享频次控制器 | 管理每日分享次数统计、分享冷却时间、防刷机制 | P1 | `GameData` |
| SH-M8 | 分享截图生成器 | 通过 Canvas 截取当前游戏画面（结算界面、主菜单）并合成为分享图片，叠加文案水印 | P2 | Canvas渲染 |

### 2.2 模块详述

#### SH-M1 分享入口调度器

管理三种分享场景的触发，每种场景具有独立的分享配置和回调行为：

| 场景ID | 来源界面 | 触发方式 | 分享类型 | 回调行为 |
|--------|---------|---------|---------|---------|
| `share_menu` | 主菜单（MainMenu） | 用户点击"分享有礼"按钮 | 好友/群 | 分享完成→判定每日限次→发放随机道具 |
| `share_win` | 胜利结算（LevelResult） | 用户点击"我要分享"按钮 | 好友/群 | 分享完成→停留在结算界面→按钮标记"已分享" |
| `share_fail` | 失败结算（LevelResult） | 用户点击"分享再战"按钮 | 好友/群 | 分享完成→自动重新开始第2关 |

调度器接收调用方传入的场景标识符 `shareScene`，选择对应的分享配置模板，调用 SH-M2 生成分享内容，然后调用 SH-M3 执行分享。

#### SH-M2 分享内容生成器

根据三种分享场景生成差异化的分享内容：

**分享文案模板**：

| 场景 | 标题（title） | 说明 |
|------|-------------|------|
| `share_menu` | "围住Boss！看你能用几步围住这个嚣张的Boss？" | 通用拉新文案，强调挑战性 |
| `share_win` | "我用{N}招就围住了Boss！不服来战！" | 炫耀成绩，N 替换为本局步数 |
| `share_fail` | "让Boss跑路了！谁来帮我围住它！" | 求助型文案，激发好友挑战欲 |

**分享图片生成策略**：

| 场景 | 图片内容 | 生成方式 |
|------|---------|---------|
| `share_menu` | 游戏Logo + 主菜单背景 + 游戏名称文字 + "来挑战"按钮装饰 | 预置静态图片（位于 `assets/share_menu.png`） |
| `share_win` | 胜利结算画面截图 + 步数/耗时水印 + "Boss被你胖揍了"文案叠加 | Canvas 实时截图合成（SH-M8） |
| `share_fail` | 失败结算画面截图 + "让Boss跑路了"文案叠加 + "帮我围住它"按钮装饰 | Canvas 实时截图合成（SH-M8） |

**分享跳转路径**（query 参数）：
- 所有分享卡片统一跳转到游戏入口路径
- 携带 scene 参数标识来源：`?src=share_menu` / `?src=share_win` / `?src=share_fail`
- 分享者的成绩参数可编码到 query 中（如 `steps=15`），供游戏启动时在加载页展示"来自好友的挑战"提示（P2功能）

#### SH-M3 好友分享执行器

封装微信小游戏 `wx.shareAppMessage` 的调用逻辑：

```typescript
// 调用方式（伪代码）
wx.shareAppMessage({
  title: shareConfig.title,      // 分享标题
  imageUrl: shareConfig.imageUrl, // 分享图片（本地路径或临时路径）
  query: shareConfig.query,       // 自定义参数（如 src=share_win&steps=15）
  // 注意：wx.shareAppMessage 需要在用户点击事件回调中同步调用
});
```

约束：
- `wx.shareAppMessage` 必须由用户主动点击触发（不能在异步回调中调用）
- 分享图片支持本地文件路径或临时文件路径（`wx.uploadFile` 返回的 URL 不适用）
- `query` 参数总长度不超过 1024 字符

#### SH-M4 朋友圈分享执行器

朋友圈分享的两种实现方案：

**方案A（P1过渡方案）**：生成分享图片后，引导用户手动保存图片到相册，然后自行发布朋友圈。通过在分享按钮旁增加"保存图片发朋友圈"的次按钮实现。

**方案B（P2完整方案）**：接入微信小游戏 `wx.shareImageToMoments` API（需微信小游戏基础库支持）。调用该 API 直接将分享图片发布到朋友圈，但该 API 目前处于内测阶段，需要微信官方权限。

#### SH-M5 分享奖励发放器

**主菜单分享奖励规则**：

1. 玩家在主菜单点击"分享有礼"按钮。
2. 分享执行（SH-M3）完成后，回调进入奖励发放器。
3. 奖励发放器检查 `GameData.dailyShareCount`：
   - 若 `dailyShareCount === 0`（今日首次分享）：执行发奖逻辑。
   - 若 `dailyShareCount > 0`（今日已分享过）：不发放奖励，弹出 Toast "今日已领取分享奖励，明天再来吧！"。
4. 发奖逻辑：
   - 从四种道具（`extra_step`, `undo`, `reset`, `hint`）中随机选取一种。
   - 调用道具系统接口，为该道具库存 +1。
   - `GameData.dailyShareCount += 1`。
   - 弹出道具获得弹窗，展示获得的道具图标和名称。
5. 主菜单分享仅在每日首次成功时发放奖励；同一天内重复分享不重复得奖，但分享本身仍然被执行（玩家可正常分享给好友）。

#### SH-M6 分享回调处理器

微信 `wx.shareAppMessage` 的回调处理逻辑：

| 回调状态 | 说明 | 处理策略 |
|---------|------|---------|
| 分享成功（用户确认发送） | 用户选择了好友/群并确认发送 | 执行该场景下的后续逻辑 |
| 分享取消（用户退出选择界面） | 用户打开分享面板但未选择任何好友直接关闭 | 不执行后续逻辑，不发放奖励 |
| 分享失败（API异常） | API调用异常（极少发生） | 记录错误日志，Toast提示"分享失败，请重试" |

**各场景后续逻辑**：

| 场景 | 成功时 | 取消时 | 失败时 |
|------|-------|-------|-------|
| `share_menu` | 检查每日限次→发放奖励→弹出道具获得弹窗 | 无操作，按钮恢复可用 | Toast提示，按钮恢复可用 |
| `share_win` | 按钮标记"已分享"（置灰3秒后恢复） | 按钮恢复可用 | 按钮恢复可用 |
| `share_fail` | 触发 `retryRequested` 事件→自动重新开始第2关 | 按钮恢复可用 | 按钮恢复可用 |

#### SH-M7 分享频次控制器

| 控制项 | 规则 | 说明 |
|-------|------|------|
| 每日分享总次数 | 不限 | 玩家可以无限次分享，但奖励仅每日首次有效 |
| 分享冷却时间 | 无 | 无硬性冷却限制，但按钮防连点 500ms 防止误触 |
| 同一分享内容重复发送 | 允许 | 同一场景多次分享发送相同内容，使用不同的时间戳区分 |
| 防刷机制 | 客户端 `dailyShareCount` 与下次启动时的校验 | 防止通过修改 localStorage 刷奖励（见 5.2） |

#### SH-M8 分享截图生成器

对于胜利和失败场景，分享图片需要包含实时游戏数据。生成流程：

1. 监听结算界面的 Canvas 渲染完成事件。
2. 将 Canvas 当前帧截取为图片数据（`canvas.toDataURL()` 或 `wx.canvasToTempFilePath`）。
3. 在离屏 Canvas 上合成最终分享图：
   - 底层：截取的结算界面截图（缩放至分享图推荐尺寸）。
   - 叠加层：游戏 Logo（左上角）、成绩数据水印（底部半透明条）、分享引导文字（"好友都在玩"等）。
4. 将合成后的 Canvas 导出为临时文件路径（`canvas.toTempFilePathSync`），传递给 SH-M3。

## 三、核心玩法规则

### 3.1 分享触发规则

| 规则编号 | 规则内容 | 例外处理 |
|---------|---------|---------|
| R1 | 分享必须由玩家主动点击按钮触发 | 任何自动化弹窗/强制分享均不允许（微信小游戏审核规范） |
| R2 | 分享按钮触发后必须在同一帧同步调用 `wx.shareAppMessage` | 不能在 setTimeout/Promise.then 等异步回调中调用 |
| R3 | 分享按钮点击后立即禁用防连点，分享完成后恢复 | 防连点窗口 500ms |
| R4 | 三种分享场景互不干扰，各自独立计数 | 胜利界面的"我要分享"不计入 `dailyShareCount` |
| R5 | 每日首次主菜单分享必然获得道具奖励 | 若玩家道具库存已满（某道具达上限），随机池排除该道具 |
| R6 | 分享图片必须清晰可辨，最小尺寸 128×128px | 微信小游戏对分享图片尺寸无硬性要求，但推荐 640×640px |

### 3.2 分享奖励规则

| 规则编号 | 规则内容 |
|---------|---------|
| R7 | 主菜单"分享有礼"的奖励发放依据 `GameData.dailyShareCount === 0` 判定 |
| R8 | 每日首次分享奖励从四种道具（extra_step, undo, reset, hint）中均匀随机选取一种 |
| R9 | 若玩家某一道具库存已达上限（10个），该道具从随机池中排除，从剩余道具中选取 |
| R10 | 若四种道具均已达上限，弹出提示"道具已满，无法获得奖励"，仍消耗每日首次分享机会 |
| R11 | 每日重置时间为 UTC+8 00:00，惰性检测（与道具系统的每日重置机制共用） |
| R12 | 胜利/失败分享不发放道具奖励，无每日次数限制 |

### 3.3 分享内容规范

| 规范项 | 要求 | 违反后果 |
|--------|------|---------|
| 分享标题 | 不超过 30 个中文字符 | 微信 API 自动截断 |
| 分享图片 | 不含诱导点击元素（虚假按钮、误导性箭头） | 微信审核拒绝/封禁分享能力 |
| 分享文案 | 不含虚假宣传（如"点击领红包"） | 微信审核拒绝 |
| 分享频次 | 不诱导用户频繁分享 | 微信小游戏运营规范 |
| 隐私合规 | 不分享用户个人数据（openid、手机号等） | 违反微信隐私政策 |

### 3.4 进入/退出条件

**分享发起条件**：
- 玩家点击任意分享入口按钮（主菜单/胜利/失败）
- 游戏已初始化微信 SDK（`wx` 对象可用）
- 当前无进行中的分享操作（防重复进入）

**分享完成条件**：
- 微信分享面板成功弹出（用户看到选择好友界面即为"已触发"）
- 用户选择好友/群并确认发送（回调 `success`）
- 或用户取消分享（回调 `fail` 且 `errMsg` 包含 "cancel"）

**分享退出条件**：
- 分享回调执行完毕（成功/取消/失败均视为退出）
- 分享按钮恢复可用状态

## 四、完整流程与状态机

### 4.1 分享系统内部状态机

```
                        ┌──────────────────────┐
                        │        IDLE          │ ← 初始状态，等待分享请求
                        └──────────┬───────────┘
                                   │ 玩家点击分享按钮
                                   v
                        ┌──────────────────────┐
                        │    CONFIGURING       │ ← 配置分享内容（文案/图片/query）
                        └──────────┬───────────┘
                                   │ 内容生成完成
                                   v
                        ┌──────────────────────┐
                        │    SHARING           │ ← 调用 wx.shareAppMessage
                        └──────────┬───────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    │              │              │
                    v              v              v
             ┌────────────┐ ┌───────────┐ ┌──────────────┐
             │ SUCCESS    │ │ CANCELLED │ │   FAILED     │
             │ (分享成功)  │ │ (用户取消) │ │ (API异常)    │
             └──────┬─────┘ └─────┬─────┘ └──────┬───────┘
                    │             │              │
                    v             v              v
             ┌──────────────────────────────────────┐
             │        REWARD / POST_PROCESS         │ ← 发奖/跳转/更新UI
             └──────────────────┬───────────────────┘
                                │ 处理完成
                                v
                        ┌──────────────────────┐
                        │        IDLE          │ ← 回到空闲状态
                        └──────────────────────┘
```

### 4.2 主菜单分享完整流程（`share_menu`）

```
[玩家进入主菜单]
       │
       ▼
[点击"分享有礼"按钮]
       │
       ▼
[按钮置为禁用态]（防连点，500ms）
       │
       ▼
[SH-M2 生成分享内容]
   ├─ title = "围住Boss！看你能用几步围住这个嚣张的Boss？"
   ├─ imageUrl = "assets/share_menu.png"（预置图片）
   └─ query = "src=share_menu"
       │
       ▼
[SH-M3 调用 wx.shareAppMessage]
       │
       ├── 成功回调：
       │    └─ [SH-M5 检查每日限次]
       │         ├─ dailyShareCount === 0：
       │         │    ├─ 随机选取一种道具
       │         │    ├─ 道具库存 +1
       │         │    ├─ dailyShareCount += 1
       │         │    └─ 弹出"获得道具"弹窗
       │         └─ dailyShareCount > 0：
       │              └─ Toast "今日已领取分享奖励"
       │
       ├── 取消回调：
       │    └─ 无操作
       │
       └── 失败回调：
            └─ Toast "分享失败，请重试"
       │
       ▼
[按钮恢复可用态]
```

### 4.3 胜利分享完整流程（`share_win`）

```
[玩家通关第2关，进入胜利结算界面]
       │
       ▼
[点击"我要分享"按钮]
       │
       ▼
[SH-M2 生成分享内容]
```

## 五、边界条件与异常处理

### 5.1 边界条件

| 编号 | 边界场景 | 预期行为 |
|------|---------|---------|
| B1 | 每日首次主菜单分享时四种道具均已达库存上限（各10个） | 弹窗提示"道具已满，无法获得奖励"，`dailyShareCount` 仍递增，不浪费每日机会 |
| B2 | 玩家在分享面板中选择好友后，发送前关闭微信分享面板（取消） | 回调标记为取消，不执行后续逻辑，按钮恢复可用 |
| B3 | 胜利界面"我要分享"分享成功后，短时间内（3秒内）再次点击 | 按钮置灰禁止点击，文案显示"已分享"，不触发新的分享 |
| B4 | 失败界面"分享再战"分享成功后，关卡切换过程中玩家强制退出 | 关卡切换为原子操作，一旦触发不可逆；玩家退出后重新进入游戏时直接进入第2关 |
| B5 | 调用 wx.shareAppMessage 时传入空字符串 title 或 imageUrl | 微信 API 自动使用默认值（游戏名称和游戏图标），但应避免此情况 |
| B6 | 玩家在分享冷却期（防连点窗口内）快速多次点击分享按钮 | 按钮在点击后立即置为禁用态，500ms 后恢复，500ms 内的后续点击被忽略 |
| B7 | Canvas 截图生成分享图片时游戏场景已被销毁（结算界面已关闭） | 截图操作在结算界面关闭前完成并缓存结果，不延迟到关闭后 |
| B8 | 同一玩家在同一天内分享后卸载重装游戏 | `dailyShareCount` 存储在 localStorage，卸载后丢失，可能导致次日首次分享无奖励；此为低概率事件，可接受 |

### 5.2 异常处理

| 编号 | 异常类型 | 处理策略 |
|------|---------|---------|
| E1 | `wx.shareAppMessage` API 不存在（微信版本过低或非微信环境） | 检测 `typeof wx.shareAppMessage === 'undefined'`，隐藏所有分享按钮；显示提示"请在微信中打开以使用分享功能" |
| E2 | `wx.shareAppMessage` 调用抛出异常 | try/catch 捕获，记录错误到日志，Toast"分享功能暂时不可用" |
| E3 | 分享图片生成失败（Canvas toTempFilePath 异常） | 降级使用预置默认分享图片（`assets/share_default.png`），不阻断分享流程 |
| E4 | 分享图片生成超时（> 1000ms） | 中断图片生成，使用预置默认图片，console.warn 记录性能告警 |
| E5 | 道具发放时 `item_system` 接口调用异常 | 记录错误到 `pipeline_errors.json`，`dailyShareCount` 回退（-1），Toast "奖励发放异常，请联系客服" |
| E6 | 玩家在分享回调处理过程中断网 | 分享成功的回调由微信客户端本地触发，不依赖网络，但道具发放后的 localStorge 写入可正常完成；无需特殊处理 |
| E7 | localStorage 中 `dailyShareCount` 数据损坏 | 读取时校验类型（`typeof === 'number'`），异常时重置为 0 |
| E8 | 防刷：玩家尝试修改 localStorage 中的 `dailyShareCount` | 每次启动游戏时校验本地记录与服务器记录（若有）；若无服务器，附加校验 `lastShareDate` 是否与当前 UTC+8 日期一致 |

### 5.3 幂等与容错

| 操作 | 幂等策略 |
|------|---------|
| 分享奖励发放 | 基于每日首次标记（`dailyShareCount === 0`），一天内只发放一次；分享回调中的奖励发放逻辑仅执行一次判断 |
| 分享成功后续动作 | 使用 `_shareCallbackHandled` 标记，防止回调被多次触发导致重复执行后续动作 |
| 分享取消/失败 | 不做任何状态修改，幂等安全 |
| Canvas 截图 | 对同一场景的多次分享请求，缓存截图结果，避免重复生成 |

## 六、前后端通信设计

### 6.1 通信概览

分享系统为纯客户端系统，所有分享操作通过微信小游戏原生 API 完成，无自建后端服务。分享系统的数据存储完全依赖于 `GameData` 单例（localStorage 持久化）。

| 数据项 | 存储位置 | 类型 | 初始值 | 说明 |
|-------|---------|------|--------|------|
| `dailyShareCount` | `GameData` | `number` | `0` | 今日主菜单分享次数（用于判断每日首次奖励） |
| `lastShareDate` | `GameData` | `string` | `""` | 最近一次分享的 UTC+8 日期（ISO格式 `"2026-05-09"`） |
| `dailyShareCount` | `GameData.progression.save_fields` | - | - | 持久化到 localStorage 的字段列表 |
| `shareImageCache` | 内存 | `Map<string, string>` | `new Map()` | 场景分享图片的临时文件路径缓存，仅内存有效，不持久化 |

**API 依赖**：

| API | 用途 | 调用模块 | 调用时机 |
|-----|------|---------|---------|
| `wx.shareAppMessage` | 分享到微信好友/群聊 | SH-M3 | 玩家点击分享按钮时 |
| `wx.shareImageToMoments` | 分享到朋友圈（P2方案B） | SH-M4 | 玩家点击"发朋友圈"时 |
| `wx.saveImageToPhotosAlbum` | 保存图片到相册（P2方案A） | SH-M4 | 玩家确认保存分享图时 |
| `wx.canvasToTempFilePath` | Canvas 截取为临时图片文件 | SH-M8 | 生成分享图片时 |
| `wx.getSetting` | 检测相册写入权限 | SH-M4 | 保存图片到相册前 |

### 6.2 事件总线协议

分享系统通过全局事件总线（EventEmitter）对外派发和监听以下事件，所有事件采用 camelCase 命名：

| 事件名 | 方向 | 载荷 | 触发时机 |
|--------|------|------|---------|
| `shareTriggered` | ShareSystem → 全局 | `{ scene: 'share_menu' \| 'share_win' \| 'share_fail', timestamp: number }` | 玩家点击任意分享按钮，分享流程启动时 |
| `shareCompleted` | ShareSystem → 全局 | `{ scene: 'share_menu' \| 'share_win' \| 'share_fail', result: 'success' \| 'cancel' \| 'fail', durationMs: number }` | 微信分享API回调返回时 |
| `shareRewarded` | ShareSystem → ItemSystem | `{ itemType: string, quantity: number, source: 'share_menu_daily' }` | 主菜单每日首次分享奖励成功发放时 |
| `shareRewardFailed` | ShareSystem → 全局 | `{ reason: 'inventory_full' \| 'api_error', itemType?: string }` | 分享奖励发放失败时（道具库存满或接口异常） |
| `shareMenuDailyDone` | ShareSystem → UI | `{ dailyShareCount: number }` | 主菜单分享且 `dailyShareCount` 从 0→1 变化时 |
| `shareImageGenerated` | ShareSystem → SH-M3 | `{ scene: string, imagePath: string, elapsedMs: number }` | 分享图片生成完成时 |
| `shareImageFailed` | ShareSystem → SH-M3 | `{ scene: string, error: string }` | 分享图片生成失败时（降级使用默认图片） |
| `shareDailyLimitHit` | ShareSystem → UI | `{ dailyShareCount: number }` | 主菜单分享但 `dailyShareCount > 0` 时 |

### 8.2 核心漏斗

| 漏斗步骤 | 环节 | 关键埋点 | 目标值 |
|---------|------|---------|--------|
| 1 | 玩家看到分享入口 | 各界面PV（主菜单/胜利/失败） | 100% |
| 2 | 玩家点击分享按钮 | `share_triggered` | 主菜单 ≥ 30%，胜利 ≥ 20%，失败 ≥ 35% |
| 3 | 微信分享面板成功弹出 | `share_completed` (result !== 'fail') | ≥ 95% |
| 4 | 用户确认发送（成功） | `share_completed` (result === 'success') | ≥ 70% |
| 5 | 分享后完成后续动作 | 道具发放 / 重开 / 标记 | ≥ 99% |

### 8.3 核心指标

| 指标 | 计算方式 | 目标值 |
|-----|---------|--------|
| 日总分享次数 | 每日 `share_completed` (success) 总数 | ≥ DAU × 0.3 |
| 分享成功率 | `share_completed (success) / share_triggered` | ≥ 70% |
| 主菜单分享渗透率 | 日点击主菜单分享用户数 / DAU | ≥ 15% |
| 胜利分享率 | 胜利界面分享次数 / 胜利局数 | ≥ 20% |
| 失败分享再战率 | 失败界面分享再战 / 失败局数 | ≥ 30% |
| 分享奖励道具分布 | 各道具类型的发放占比 | 均匀分布（各 25%） |
| 分享卡片点击转化率 | 通过分享卡片进入游戏 / 分享卡片曝光 | ≥ 5%（需微信分享统计支持） |

### 9.5 明确不做

| 事项 | 不做原因 |
|------|---------|
| 强制分享（分享后才能继续游戏）| 违反微信小游戏审核规范，且损害用户体验 |
| 虚假分享奖励（分享后实际不发放奖励）| 违反微信运营规范，损害玩家信任 |
| 分享后读取好友关系链数据（如获取好友列表）| 隐私合规风险，微信 API 限制 |
| 非微信渠道分享（QQ、微博等）| "围住Boss"为微信小游戏，仅支持微信生态内分享 |
| 分享内容中包含用户真实姓名/手机号 | 严重隐私违规 |
| 分享链接直接跳转到 App Store 或第三方下载 | 微信小游戏禁止外跳下载 |
| 基于分享次数的排行榜 | 鼓励无意义刷分享，破坏游戏公平性 |
