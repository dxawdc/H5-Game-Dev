# 核心玩法系统 系统策划案

**文档版本**：v1.0  **系统ID**：gameplay_core  **优先级**：P0  **复杂度**：L
**所属游戏**：围住Boss

---

## 一、定位与目标

### 1.1 系统存在的理由

核心玩法系统是"围住Boss"整个游戏体验的基石，负责承载玩家与游戏世界之间最基础的交互闭环。玩家通过点击网格交点放置障碍物，Boss利用A*寻路算法试图逃脱——这套"放置-移动-反馈"的循环构成游戏的全部核心乐趣。该系统决定了游戏的策略深度、手感节奏和长期可玩性。没有这套系统，游戏将不复存在。

具体而言，本系统解决以下三个核心问题：

1. **规则闭环**：定义"围住Boss"这一核心概念的完整判定逻辑，确保每一局游戏有明确的胜负标准和可复现的规则体系。
2. **交互反馈**：将玩家的触控点击转化为有意义的游戏操作，每步操作后Boss以可视化的方式做出反应，形成清晰的因果链。
3. **难度曲线**：通过网格尺寸（7×7教程关、9×9标准关、11×11困难关）和Boss AI参数（搜索深度、启发函数权重）的组合调节，实现从入门到精通的渐进式难度。

### 1.2 核心目标指标

| 目标维度 | 具体贡献 | 衡量方式 |
|---------|---------|---------|
| 新手上手率 | 教程关前3步内理解"障碍物堵路"的核心逻辑 | 教程关完成率 ≥ 85%（埋点：tutorialCompleted） |
| 单局完成度 | 玩家不会在中途因困惑而放弃 | 单局放弃率 ≤ 5%（埋点：gameAbandoned） |
| 策略深度 | 存在多条不同思路的通关路径 | 天梯高分玩家平均思考时间 ≥ 3秒/步（埋点：stepThinkTime） |
| 付费转化 | 道具在关键时刻提供破局价值 | 道具使用率 ≥ 40%玩家在通关失败3次后使用道具 |
| 回访粘性 | 玩家愿意反复挑战更高关卡 | 日活跃玩家平均每日对局数 ≥ 5局 |

### 1.4 设计红线

- **公平性原则**：Boss不可以使用任何玩家无法获知的信息做决策（不允许"透视"玩家未放置的障碍物）。A*寻路基于当前棋盘可见状态计算，不允许作弊偏移。
- **原子操作原则**：玩家每步操作 = 一个障碍物放置 + Boss移动一步，不允许玩家连续放置、也不允许Boss连续移动，维持回合制公平。
- **触控友好原则**：所有可点击的网格交点区域最小 88×88 逻辑像素，确保手机触控无误触。
- **状态可回溯**：游戏状态必须可序列化，支持"撤销"道具将棋盘恢复至上一步的完整状态快照（含障碍物布局和Boss坐标）。
- **无随机性原则**：给定同样的棋盘状态和同样的Boss起始位置，Boss的寻路结果必须完全确定。不允许在寻路中引入随机因子。

## 三、核心玩法规则

### 3.1 基础规则

1. **地图与网格**：棋盘由 N×N 个格子组成，实际可点击位置为 (N+1)×(N+1) 个网格交点。默认 N=7（教程关）或 N=9（标准关）。
2. **Boss出生**：每个关卡Boss从指定的内部坐标点出发（不在地图边界上）。
3. **玩家操作**：玩家每次点击一个未被占用的网格交点，在该交点放置一个障碍物（视觉表现为一个木栅栏或石墩）。
4. **Boss移动**：玩家每次放置障碍物后，Boss沿网格线朝出口方向移动一格（上/下/左/右四个方向，不允许对角线移动）。移动路径由A*算法计算得出。
5. **障碍物规则**：障碍物放置在网格交点上，会阻断经过该交点的所有路径。Boss不能通过有障碍物的交点。
6. **出口规则**：地图的四个边界均为可能的逃脱出口。Boss一旦移动到边界交点（即最外围的网格交点），判定为逃脱成功，玩家失败。
7. **围住判定**：当Boss所在位置的上下左右四个相邻交点均被障碍物占据，或所有可达的交点均被障碍物围堵且Boss无法到达任一出口时，判定为围住成功，玩家胜利。
8. **步数限制**：标准关卡默认玩家有 30 步（N=9）或 18 步（N=7）的限制。步数耗尽未获胜则判定玩家失败。部分关卡无步数限制（无限步模式）。

### 3.2 数值规则表

| 参数名 | 教程关 | 简单关 | 标准关 | 困难关 | 说明 |
|--------|-------|--------|--------|--------|------|
| 网格尺寸 N | 7 | 7 | 9 | 11 | 格子数量 N×N |
| 交点总数 | 64 | 64 | 100 | 144 | (N+1)×(N+1) |
| 初始障碍物数 | 4 | 2 | 0~3 | 0~5 | 关卡预置（教程关强制示例布局） |
| 最大步数 | 99（无限制）| 20 | 30 | 40 | 玩家可用步数上限 |
| Boss初始网格距离 | 3 | 3~4 | 4~5 | 5~6 | Boss出生点到最近出口的曼哈顿距离 |
| A*启发函数权重 | 1.2 | 1.5 | 1.8 | 2.0 | 权重越高，"贪心"倾向越强 |
| A*搜索深度上限 | 100 | 200 | 500 | 2000 | 防止极端棋盘下计算过载 |
| 获胜路径推荐数 | 1 | 1~2 | 1~3 | 2~4 | 关卡设计时预设的可行获胜路径数 |

### 3.3 获取/消耗规则

| 资源类型 | 获取方式 | 消耗方式 | 上限 |
|---------|---------|---------|------|
| 步数 | 关卡开始时赋予（按关卡配置）| 每放置一个障碍物消耗 1 步 | 按关卡配置（最大99）|
| 额外步数道具 | 商城购买/每日奖励/关卡掉落 | 使用后当前关卡通用水步数 +5 | 每局限用 2 次 |
| 撤销道具 | 商城购买/每日奖励 | 使用后回退一步（含Boss移动）| 每局限用 3 次 |
| 重开道具 | 商城购买/每日奖励 | 使用后当前关卡完全重置 | 不限制 |
| 提示道具 | 商城购买/通关奖励 | 使用后高亮显示推荐的下一个放置点 | 每局限用 1 次 |

### 3.4 进入/退出条件

**进入条件**：
- 玩家从关卡选择界面点击任一未通关或已解锁的关卡
- 系统加载该关卡配置数据，初始化棋盘状态，放置预置障碍物，设置Boss出生点
- 进入"准备"（ready）状态，等待玩家首次操作

**退出条件（正常）**：
- 胜利退出：Boss被完全围住 → 播放胜利动画 → 结算界面 → 返回关卡选择
- 失败退出（逃脱）：Boss到达边界 → 播放失败动画 → 结算界面 → 可选择重试或返回
- 失败退出（步数耗尽）：步数归零且Boss未被围住 → 结算界面 → 可选择重试或返回

**退出条件（异常）**：
- 玩家主动点击暂停 → 暂停菜单 → 选择"放弃本局" → 确认弹窗 → 退出并返回关卡选择
- 网络断开或应用切后台 → 自动保存当前状态快照 → 重新进入时恢复

## 四、完整流程与状态机

### 4.1 主流程

```
[关卡加载] → 读取关卡配置 → 初始化棋盘 → 放置预置障碍物 → 设置Boss
     ↓
[ready状态] → 等待玩家首次点击
     ↓
[玩家点击交点] → 验证是否有效（未占用、在边界内、未结束）
     ↓
[无效操作] → 播放轻触反馈（震动/音效） → 回到等待
     ↓
[有效操作] → 放置障碍物 → 扣除步数
     ↓
[Boss移动阶段] → A*计算下一格 → Boss移动到目标点
     ↓
[胜负判定] → 检查Boss是否在边界 → 检查Boss是否被围住 → 检查步数
     ↓
[游戏未结束] → 回到等待（playerTurn状态）
     ↓
[游戏结束] → 播放结算 → 数据上报 → 回到关卡选择
```

### 4.2 状态定义

| 状态名 | 状态ID | 允许操作 | 说明 |
|--------|--------|---------|------|
| 加载中 | `loading` | 无 | 关卡数据加载中，展示加载动画 |
| 准备 | `ready` | 无（展示开场动画后自动转入playerTurn）| Boss就位，开场动画播放 |
| 玩家回合 | `playerTurn` | 点击网格交点 | 等待玩家放置障碍物 |
| Boss移动 | `bossMoving` | 无 | Boss沿网格线移动，播放移动动画，期间屏蔽所有操作 |
| 胜负判定 | `judging` | 无 | 判定器运行中，短暂延时后出结果 |
| 胜利 | `victory` | 点击"继续" | Boss被围住，胜利动画播放 |
| 失败-逃脱 | `defeatEscaped` | 点击"重试"或"返回" | Boss到达边界逃脱 |
| 失败-步数耗尽 | `defeatOutOfSteps` | 点击"重试"或"返回" | 步数为零未获胜 |
| 暂停 | `paused` | 点击"继续"、"重开"、"返回" | 暂停菜单覆盖层 |
| 道具使用 | `itemUsing` | 选择目标（撤销/提示/额外步数/重开）| 道具选取中 |

**状态转换表**：

```
loading → ready（数据加载完成）
ready → playerTurn（开场动画结束）
playerTurn → bossMoving（玩家放置障碍物成功）
playerTurn → paused（玩家点击暂停按钮）
bossMoving → judging（Boss移动动画结束）
judging → playerTurn（胜负均未达成）
judging → victory（Boss被围住）
judging → defeatEscaped（Boss到达边界）
judging → defeatOutOfSteps（步数耗尽）
paused → playerTurn（点击"继续"）
paused → loading（点击"重开"）
playerTurn → itemUsing（使用道具）
itemUsing → playerTurn（道具使用完成/取消）
victory → levelSelect（结算展示完毕）
defeatEscaped → levelSelect（结算展示完毕）
defeatOutOfSteps → levelSelect（结算展示完毕）
defeatEscaped → loading（点击"重试"）
defeatOutOfSteps → loading（点击"重试"）
```

### 4.3 异常流程

1. **Boss无路可走但未到边界**：
   - A*返回空路径 → 判定器直接判定为围住成功 → 转入 `victory` 状态
   - 此时Boss不移动，直接播放"被围住"的特殊动画

2. **玩家连续快速点击**：
   - 在 `playerTurn` 状态下，首次有效点击后立即进入 `bossMoving`，后续点击被状态机拦截（输入屏蔽层）
   - 回调防抖：连续点击间隔 < 200ms 的点击事件只取最后一次

3. **使用撤销时无历史记录**：
   - 玩家在第一步操作前点击撤销 → 弹出Toast提示"没有可撤销的步骤"
   - 撤销队列为空时，撤销按钮置灰

4. **步数耗尽与Boss移动同时发生**：
   - 玩家放置最后一个障碍物后步数为0，此时仍需等待Boss移动完成再做判定
   - Boss移动完成后，先检查胜负条件，再检查步数（胜利优先于步数耗尽）

5. **A*搜索超时**：
   - 设定50ms最大计算时间，超时后降级为BFS搜索，保证游戏不卡死
   - 上报性能监控埋点（aStarTimeout）

## 五、边界条件与异常处理

### 5.1 数值边界

| 边界场景 | 预期行为 | 防御措施 |
|---------|---------|---------|
| 棋盘已满（全部交点被占）| 判定Boss是否被围住，立即结算 | 放置前检查剩余空位，空位为0时不允许再放 |
| 步数为1且棋盘远未封闭 | 玩家放置后步数归零，Boss移动后按胜负→步数顺序判定 | 步数达到1时，UI层闪烁提示"最后一步" |
| Boss坐标与障碍物重叠 | 不可能发生（障碍物只能放置在交点上，Boss在格子里） | 坐标系分离：交点用(x,y)，格子用[x,y]，转换时做边界校验 |
| 连续撤销直至回到第一步 | 棋盘恢复初始状态，步数恢复为初始值 | 撤销队列最多保留 30 条历史的完整状态快照 |
| 额外步数道具叠加使用 | 回合步数上限+5，可叠加最多2次共计+10 | 计数器上限校验，超过上限弹出"已达上限"提示 |

### 5.2 非法操作拦截

| 非法操作 | 拦截层级 | 反馈方式 |
|---------|---------|---------|
| 点击已被障碍物占据的交点 | UI层 | 轻微震动 + 音效，交点高亮变红 0.3秒后恢复 |
| 在bossMoving状态点击棋盘 | 状态机层 | 全部点击事件被屏蔽层拦截，不传递到GameCore |
| 在judging状态使用道具 | 逻辑层 | 道具面板置灰不可点击 |
| 步数耗尽后继续放置 | 逻辑层 | 自动弹出结算界面，不响应放置请求 |
| 点击地图范围外的坐标 | UI层 | 无任何反馈（点击判定超出碰撞区域） |
| 反复快速使用同一道具 | 逻辑层 | 道具使用后进入500ms冷却，冷却期内按钮不可用 |
| 网络请求未返回时重复请求 | 网络层 | 请求ID去重，相同requestId的请求在pending期内忽略 |

### 5.3 幂等设计

| 操作类型 | 幂等key | 实现方式 |
|---------|--------|---------|
| 步数扣除 | `roundId + stepIndex` | 每个对局回合有唯一stepIndex，服务端判定stepIndex是否已使用 |
| Boss移动 | `roundId + stepIndex` | 同一stepIndex只执行一次Boss移动计算 |
| 胜负判定 | `roundId` | 单局游戏只生成一次胜负结果，即使重复上报也以第一次为准 |
| 道具使用 | `roundId + itemType + timestamp` | 使用记录写入本地和远端，重放时跳过已使用记录 |
| 埋点上报 | `eventId` | 每个埋点事件有唯一eventId，服务端去重 |
| 状态快照 | `roundId + stepIndex` | 单步快照写入后不可变，覆盖写入需校验原快照是否为空 |

## 六、前后端通信设计

### 6.1 接口清单

| 接口名 | 方向 | 请求参数 | 响应参数 | 说明 |
|--------|------|---------|---------|------|
| `gameStart` | 客户端→服务端 | `{ levelId: string, timestamp: number }` | `{ roundId: string, boardState: BoardSnapshot, bossPosition: Position, totalSteps: number }` | 开始对局，服务端创建游戏记录 |
| `placeObstacle` | 客户端→服务端 | `{ roundId: string, position: Position, stepIndex: number }` | `{ success: boolean, newBossPos: Position, boardState: BoardSnapshot, stepsLeft: number, gameResult: string \| null }` | 放置障碍物并触发Boss移动 |
| `useItem` | 客户端→服务端 | `{ roundId: string, itemType: string, target?: Position }` | `{ success: boolean, boardState: BoardSnapshot, stepsLeft: number, itemsRemaining: Record<string, number> }` | 使用道具 |
| `giveUp` | 客户端→服务端 | `{ roundId: string }` | `{ success: boolean }` | 放弃当前对局 |
| `getHint` | 客户端→服务端 | `{ roundId: string }` | `{ hintPosition: Position, pathPreview: Position[] }` | 获取下一步推荐放置点 |
| `gameResult` | 服务端→客户端 | 无（服务端主动推送）| `{ roundId: string, result: 'victory' \| 'defeatEscaped' \| 'defeatOutOfSteps', stepsUsed: number, itemsUsed: string[], score: number }` | 最终对局结果推送 |

### 6.2 关键事件/信号

所有事件使用 `camelCase` 命名，通过全局事件总线派发。

| 事件名 | 方向 | 载荷 | 触发时机 |
|--------|------|------|---------|

### 9.4 明确不做

| 事项 | 不做原因 | 备注 |
|------|---------|------|
| 有时间限制的模式（限时竞速） | 核心玩法是回合制策略，引入实时计时破坏游戏本该有的"思考"体验 | 如有需求建议做独立子游戏 |
| Boss使用复杂AI（如蒙特卡洛树搜索）| 休闲游戏的Boss不需要"太聪明"，A*已足够提供挑战；复杂AI会让玩家感到挫败而不是有趣 | 保持Boss"可预测的聪明" |
| 联机实时对战 | 技术复杂度高（需要同步帧）、用户基数和付费模型不支持 | 异步分享谜题已够用 |
| 3D化/2.5D视角 | 手机端3D渲染性能消耗大，且会干扰网格点击的精度 | 坚持2D俯瞰视角 |
| 剧情/角色养成 | 围住Boss的核心卖点是益智解谜，不需要剧情包装或角色升级 | 专注谜题设计本身 |
| 内嵌广告（激励视频除外）| 横幅或插屏广告严重破坏沉浸感 | 仅保留激励视频广告位（复活/获取道具）|
| 无限关卡自动生成（无人工校验）| 自动生成的关卡无法保证有唯一解且质量可控 | 所有关卡需经过策划手动测试验证 |
