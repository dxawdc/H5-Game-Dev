# 关卡结算系统 系统策划案

**文档版本**：v1.0  **系统ID**：level_result  **优先级**：P0  **复杂度**：S
**所属游戏**：围住Boss（Surround the Boss）
**对应脚本**：LevelResultScreen.ts  **模块类型**：Scene

---

## 一、定位与目标

### 1.1 系统定位

关卡结算系统是玩家完成（通关或失败）一局游戏后进入的过渡界面，承担"承上启下"的枢纽角色。它是玩家每局游戏的情感终点——胜利时的成就感在这里被量化并放大，失败时的挫败感在这里被引导转化为再战动力。结算系统在留存循环中处于"回顾结果 → 触发下一步行动"的关键位置，直接影响玩家的短期回访行为和社交传播意愿。

针对"围住Boss"的休闲卡通关卡结构，结算系统需要处理两种截然不同的关卡：第1关（教学关，7×7网格）自动衔接第2关，不展示结算界面；第2关（正式挑战，9×9网格）分别处理通关和失败两种结果，每种结果匹配不同的视觉风格、文案情绪和操作选项。

### 1.2 核心目标

1. **情绪放大**：胜利时通过视觉和文案强化玩家成就感，让"围住Boss"的爽感被量化感知（几步、几秒）
2. **失败转化**：失败时通过逗趣的Boss表情和文案缓解挫败感，将负面情绪转化为"再试一次"的动力
3. **社交裂变**：通过"我要分享"和"分享再战"按钮驱动微信小游戏的社交传播，分享结果截图自带炫耀/吐槽属性
4. **留存驱动**：提供"继续挑战"（重试关卡2）和"一会重来"（返回主界面）选项，让玩家始终保持低门槛的下一步路径
5. **商业化铺垫**：为未来的广告复活/激励视频广告位预留接口（"分享再战"按钮在P1后可改为"看广告复活"再战的入口）

### 1.3 与上下游系统的关系

| 系统 | 关系 |
|-----|------|
| 核心玩法系统（GameScene） | 上游/下游：玩法系统在通关/失败时触发结算，传入结果数据（步数、耗时、是否通关）；结算界面触发重试时通知玩法系统重新初始化对局 |
| 主界面系统（MainMenu） | 下游：结算完成后根据玩家操作跳转到主界面或直接重启关卡 |
| 关卡系统（LevelConfig） | 读取关卡配置，区分第1关和第2关的行为逻辑 |
| GameData单例 | 结算时写入通关记录（步数、耗时、通关次数），读取历史最佳成绩用于对比 |
| 分享系统 | 调用微信分享API，将结算结果分享到好友/群聊 |
| 广告系统（P1/P2） | 失败时"看广告复活"入口，通关后激励视频广告入口 |
| 道具系统（ItemSystem） | 上游：读取道具库存信息，在结算界面提供道具使用选项（如道具消耗重试） |

### 1.4 玩家画像与使用场景

- **第1关通关玩家**：第1关（教学关）通关后自动进入第2关，不经结算界面，体验无中断的"教学→实战"流程
- **第2关胜利玩家**：成功围住Boss，看到Boss鼻青脸肿的滑稽表情，获得步数和耗时数据，可选择分享炫耀或继续挑战
- **第2关失败玩家**：Boss成功逃脱，看到Boss得意的嘲讽表情，可选择分享吐槽或返回主界面冷静
- **重复挑战玩家**：多次挑战第2关的玩家，每次结算时可以看到本次成绩与历史最佳的对比（P2功能）
- **社交分享玩家**：主动或被动触发分享，将结算结果图分享到微信好友/群，吸引好友点击进入游戏

### 2.2 M1 结算入口判定

**职责**：接收核心玩法系统派发的结算事件，判断是否展示结算界面以及展示哪种类型。

**判定规则**：
1. 读取传入的结果数据 `{ levelId: number, isWin: boolean, steps: number, timeSeconds: number }`
2. 若 `levelId === 1`（第1关）且 `isWin === true`：不展示结算界面，直接触发场景切换到第2关
3. 若 `levelId === 2`（第2关）且 `isWin === true`：展示胜利结算界面（M2）
4. 若 `levelId === 2`（第2关）且 `isWin === false`：展示失败结算界面（M3）
5. 若 `levelId === 1` 且 `isWin === false`（教学关失败）：返回第1关重试（不展示结算界面）

### 2.3 M2 胜利界面

**视觉布局**（自上而下）：

1. **背景层**：全屏暗色遮罩（rgba 0,0,0,0.6）上叠加暖色庆祝光效（金色粒子/射线光晕，500ms入场动画）
2. **Boss角色**：屏幕中上方（y ≈ 20%~35%），显示Boss鼻青脸肿（肿眼泡、贴创可贴、转圈星星）的静态或轻微动画表情，尺寸约120×120px
3. **胜利标题**：Boss下方，大字标题文本，字号28px，颜色金色 `#FFD700`，粗体，带描边
   - 主文案："Boss被你胖揍了！"
   - 备选文案池（随机展示，增加趣味性）：
     - "Boss被你围哭了！"
     - "Boss求饶了！"
     - "大获全胜！"
4. **统计信息**：标题下方，展示通关数据面板
   - 格式："搞定Boss，只用{N}招，不到{M}秒"
   - N = 本次通关步数，M = 本次通关耗时（向上取整秒数）
   - 展示历史最佳对比（P2）：若本次步数/时间低于历史最佳，显示"新纪录！"标签
5. **操作按钮组**：屏幕下方区域（y ≈ 65%~85%）

**按钮**（M5详细设计）：
- **"继续挑战"**：重新开始第2关，关卡难度维持不变，重置网格状态
- **"我要分享"**：触发分享功能（M6），分享当前胜利结果
- 两个按钮水平并列，"继续挑战"居左（主色调按钮），"我要分享"居右（次色调按钮）

**入场动画**：
- 遮罩层淡入（0→0.6，300ms）
- Boss图标从底部弹入（400ms ease-out bounce）
- 标题文本逐字打印效果（每字100ms间隔）
- 统计信息从左右两侧滑入（各300ms，间隔100ms）
- 按钮组淡入（400ms，延迟200ms）

### 2.5 M4 统计信息展示

**数据结构**：
```typescript
interface LevelResultStats {
  steps: number;          // 本局步数
  timeSeconds: number;    // 本局耗时（秒）
  isNewRecord: boolean;   // 是否刷新历史记录（P2）
  bestSteps: number;      // 历史最佳步数（P2）
  bestTime: number;       // 历史最佳耗时（P2）

```

**展示规则**：
- 胜利界面：完整展示步数和耗时，步数为主数据（大字），耗时为副数据（小字）
- 失败界面：不展示步数/耗时（失败无意义），只展示吐槽文案
- P2版本：新增历史最佳对比，若本次通关步数 ≤ 历史步数记录，显示星标 "新纪录！"

### 2.7 M6 分享功能

**触发路径**：
- 胜利界面点击"我要分享"：触发 `wx.shareAppMessage`，分享胜利结果图，分享后不跳转，停留在结算界面
- 失败界面点击"分享再战"：触发 `wx.shareAppMessage`，分享失败吐槽图，分享完成后自动重新开始第2关

**分享配置**：
```
分享标题：胜利时 "我用{N}步围住了Boss！" / 失败时 "让Boss跑路了，谁来帮我？"
分享图片：Canvas截图生成当前结算界面（含Boss表情和成绩）
分享路径：pages/index/index（游戏入口路径）
```

**分享后逻辑**：
- "我要分享"：分享回调后停留在结算界面，按钮文案变为"已分享"（禁用状态，3秒后恢复）
- "分享再战"：分享回调后立即触发 `levelRestarted` 事件，切换到游戏关卡场景

## 三、核心玩法规则

### 3.1 关卡1自动跳过关卡结算规则

第1关（教学关）是引导玩家掌握基础操作的特殊关卡，其结算逻辑不同于第2关：

- **第1关通关**：玩家完成教学步骤（围住Boss）后，不展示结算界面，由核心玩法系统直接触发场景切换至第2关。切换过程展示简短的过场提示（可选，500ms文字提示"进入第2关！"）
- **第1关失败**：教学关通常设计为不可能失败（有引导提示），若玩家异常操作导致失败（如主动退出），直接重开第1关，不展示结算界面

此设计决策基于以下考虑：
- 保持教学关到正式关卡的流畅体验，避免中断玩家的学习心流
- 第1关作为教学关，结果不具备统计意义（步数和耗时不计入历史记录）
- 减少结算系统对新手流程的干扰

### 3.2 统计数据显示规则

胜利界面的统计信息遵循以下规则：

1. **步数显示**：使用玩家在关卡中实际生效的步数（道具"返回上1步"使用的步数不回溯，以最终提交时的总步数为准）
2. **耗时显示**：从关卡场景加载完成（`sceneTransitionComplete`）到Boss被围住的实时耗时，单位为秒，向上取整
3. **步数单位**：显示为"招"，如"只用5招"（符合"围住Boss"的武侠/搞笑混搭风格）
4. **耗时单位**：显示为"秒"，如"不到12秒"，始终向上取整，最小值为1秒
5. **格式模板**："搞定Boss，只用{N}招，不到{M}秒"
   - 若步数 ≤ 3：特殊文案 "秒杀！只用{N}招！"
   - 若耗时 ≤ 5秒：特殊文案 "速通！不到{M}秒！"
   - 若步数 > 历史步数 × 1.5（P2）：文案变为 "虽然慢了点儿，但还是赢了！"

### 3.3 失败界面的特殊规则

失败界面不展示具体步数和耗时（失败局的数据无正面意义），仅展示吐槽文案。原因：
- 避免玩家因"差一步就能赢"之类的数据产生额外挫败感
- 保持界面的简洁和统一，让玩家注意力集中在"下一步行动"（重试或返回）
- 步数/耗时数据仅在胜利时作为成就感量化的工具

未来版本（P2）可以考虑在失败界面展示"最佳距离"（Boss逃脱时距出口的步数），作为隐性的"你离胜利很近"的心理暗示。

### 3.4 按钮状态管理

- 在分享API调用期间，所有按钮进入禁用状态（`disabled`），防止玩家连点导致多次调用
- 分享回调完成后，根据按钮类型决定恢复行为：
  - "我要分享"：恢复启用，但文案改为"已分享"并置灰3秒作为反馈
  - "分享再战"：不恢复，直接触发场景切换
- 按钮防连点窗口期：200ms（点击后200ms内不响应第二次点击）

## 四、完整流程与状态机

### 4.1 状态机定义

```
                     ┌─────────────────┐
                     │     IDLE        │
                     └────────┬────────┘
                              │ 收到结算事件
                              v
                     ┌─────────────────┐
                     │  ENTERING       │ ← 入场动画播放中
                     └────────┬────────┘
                              │ 入场动画完成
                              v
                     ┌─────────────────┐
                     │  DISPLAYING     │ ← 展示结算信息，等待玩家操作
                     ├───────┬─────────┤
                     │       │         │
                     │   胜利模式   失败模式
                     └───┬───┴────┬────┘
                         │        │
              ┌──────────┤        ├──────────┐
              v          v        v          v
         SHARING    RETRY    SHARING    BACK_HOME
              │          │        │          │
              v          v        v          v
         ┌─────────────────────────────────────┐
         │         TRANSITIONING               │ ← 场景切换中
         └────────────────┬────────────────────┘
                          │
                          v
                  ┌─────────────────┐
                  │   EXIT (场景)    │
                  └─────────────────┘
```

### 4.2 详细流程描述

#### 流程A：胜利结算流程

1. 核心玩法系统判定Boss被完全围住，触发 `levelCleared` 事件
2. 结算系统 `M1` 接收事件，判断 `levelId === 2`，进入胜利结算流程
3. 系统进入 `ENTERING` 状态：
   - 关卡场景隐藏（或销毁）
   - 渲染结算遮罩层（暗色渐变背景）
   - 执行胜利界面入场动画（Boss弹入 → 标题逐字 → 统计滑入 → 按钮淡入）
   - 动画总时长约 1200ms
4. 入场动画完成，进入 `DISPLAYING`（胜利模式）状态：
   - Boss展示鼻青脸肿表情
   - 展示胜利标题（随机从文案池选取）
   - 格式化统计文本："搞定Boss，只用{N}招，不到{M}秒"
   - 两个按钮可用："继续挑战" + "我要分享"
5. 玩家操作分支：
   - **点击"继续挑战"** → 触发 `retryRequested` 事件 → 进入 `TRANSITIONING` → 场景切换到第2关
   - **点击"我要分享"** → 进入 `SHARING` 状态 → 调用微信分享API → 回调后回到 `DISPLAYING`（按钮标记"已分享"）

#### 流程B：失败结算流程

1. 核心玩法系统判定Boss到达出口，触发 `levelFailed` 事件
2. 结算系统 `M1` 接收事件，判断 `levelId === 2`，进入失败结算流程
3. 系统进入 `ENTERING` 状态：
   - 关卡场景隐藏
   - 渲染结算遮罩层
   - 执行失败界面入场动画（Boss掉入 → 标题划入 → 吐槽淡入 → 按钮淡入）
   - 动画总时长约 1000ms
4. 入场动画完成，进入 `DISPLAYING`（失败模式）状态：
   - Boss展示得意表情（小幅左右摇摆动画）
   - 展示失败标题（随机从文案池选取）
   - 展示吐槽文案（随机从文案池选取）
   - 两个按钮可用："一会重来" + "分享再战"
5. 玩家操作分支：
   - **点击"一会重来"** → 触发 `returnedToMainMenu` 事件 → 进入 `TRANSITIONING` → 场景切换到主界面
   - **点击"分享再战"** → 进入 `SHARING` 状态 → 调用微信分享API → 回调后自动触发 `retryRequested` → 进入 `TRANSITIONING` → 场景切换到第2关

#### 流程C：第1关通关跳关流程

1. 核心玩法系统判定Boss被围住，触发 `levelCleared` 事件，带入 `{ levelId: 1, isWin: true, steps: N, timeSeconds: M }`
2. 结算系统 `M1` 接收事件，检测到 `levelId === 1`：
   - 不进入 `ENTERING` 状态
   - 直接触发 `level2Entered` 事件
   - 可选：展示一个短暂（500ms）的过场提示"进入第2关！"
3. 场景切换到第2关（`GameLevel` 场景，关卡ID=2）

### 4.3 场景切换协议

```typescript
interface ILevelResultTransition {
  source: 'victory' | 'defeat';
  action: 'retry_level_2' | 'back_to_main' | 'share_retry' | 'share_only';
  animation: 'fade_out';
  duration: 300;          // ms
  onComplete: () => void; // SceneManager.switchTo 调用入口
}
```

## 五、边界条件与异常处理

### 5.1 边界条件

| 编号 | 条件 | 预期行为 |
|------|------|---------|
| B1 | 第2关通关时步数为0（异常情况，理论上不可能） | 步数显示为"--"，文案变为"搞定Boss，神不知鬼不觉！" |
| B2 | 第2关通关耗时不足1秒 | 耗时显示"不到1秒"，文案变为"瞬杀！" |
| B3 | 玩家在入场动画播放期间快速点击按钮 | 按钮在入场动画期间全部禁用，动画完成后才启用 |
| B4 | 第2关首次通关（历史无记录） | 统计信息正常显示本次数据，不展示历史对比 | 
| B5 | 同一角色连续快速通关/失败 | 每次结算独立渲染，前一次结算界面完全销毁后再创建新实例 |
| B6 | 分享API被用户拒绝（取消分享） | 分享回调仍会触发（微信API行为），按分享成功处理（保持现有逻辑） |
| B7 | 关卡2失败后返回主界面，再进入关卡2，再失败 | 按正常失败流程重复，无特殊限制，不限制失败次数 |
| B8 | 关卡1通关数据异常（非教学关应有的步数） | 关卡1数据不做校验，直接跳过结算进入关卡2 |

### 5.2 异常处理

| 编号 | 异常类型 | 处理策略 |
|------|---------|---------|
| E1 | 结算事件数据格式异常（缺失steps/timeSeconds） | 使用默认占位值（steps: "--", timeSeconds: "--"），console.warn 提示 |
| E2 | 微信分享API调用失败 | 静默处理，按钮恢复可用，"分享再战"降级为直接重开（不分享） |
| E3 | 结算资源（Boss表情图）加载失败 | 使用文字表情替代（如"[鼻青脸肿]"或"[得意]")，不阻断流程 |
| E4 | 场景切换过程中异常中断 | 设置场景切换超时（5秒），超时后尝试重新切换，若仍失败则强制跳转到主界面 |
| E5 | localStorage写入失败（记录成绩时） | 静默失败，不阻断结算流程，下次启动时游戏状态可能回退 |

### 5.3 错误恢复策略

- 分享API失败（E2）：不展示失败提示，按钮恢复正常可用状态，玩家可再次尝试分享
- 场景切换失败（E4）：5秒超时后重试一次，重试仍失败则调用 `wx.navigateBack()` 回退
- 数据异常（E1、E5）：不做阻断式提示，使用默认值兜底确保界面正常展示

## 六、前后端通信设计

### 6.1 通信概览

关卡结算系统属于纯客户端系统，无服务端实时通信。结算数据通过全局事件总线接收（来自核心玩法系统），处理结果（通关记录）持久化写入本地的 `GameData` 单例。

排行榜数据不依赖结算系统直接写入，由主界面系统在玩家返回主界面后统一刷新。

### 6.2 事件总线协议

结算系统通过全局事件总线（EventEmitter）收发事件，所有事件名采用 camelCase 命名：

**传入事件（结算系统监听）**：

| 事件名 | 来源 | 载荷 | 触发时机 |
|--------|------|------|---------|
| `levelCleared` | 核心玩法系统 | `{ levelId: number, steps: number, timeSeconds: number }` | 玩家成功围住Boss时触发 |
| `levelFailed` | 核心玩法系统 | `{ levelId: number, steps: number, timeSeconds: number }` | Boss成功逃脱时触发 |

**传出事件（结算系统派发）**：

| 事件名 | 方向 | 载荷 | 触发时机 |
|--------|------|------|---------|
| `levelCleared`（转发） | 结算 → EventBus | `{ levelId: number, steps: number, timeSeconds: number, isNewRecord: boolean }` | 胜利结算展示完成后派发（供埋点模块监听） |
| `levelFailed`（转发） | 结算 → EventBus | `{ levelId: number, steps: number, timeSeconds: number }` | 失败结算展示完成后派发（供埋点模块监听） |
| `shared` | 结算 → 分享系统 | `{ resultType: 'victory' | 'defeat', steps?: number, timeSeconds?: number }` | 玩家点击任一分享按钮时触发 |
| `retryRequested` | 结算 → 场景管理器 | `{ levelId: 2 }` | 玩家点击"继续挑战"或"分享再战"（分享完成后）时触发 |
| `returnedToMainMenu` | 结算 → 场景管理器 | `{}` | 玩家点击"一会重来"时触发 |
| `level2Entered` | 结算 → 场景管理器 | `{}` | 第1关通关后自动进入第2关时触发（跳过结算界面） |

### 6.3 数据存储设计

结算系统在胜利时向 `GameData` 写入以下字段：

```typescript
// 结算系统相关存储字段（GameData 单例）
interface LevelResultStorageData {
  // 第2关通关记录
  level2Cleared: boolean;            // 是否曾经通关第2关
  totalClears: number;               // 第2关累计通关次数（用于排行榜）
  bestLevel2Steps: number;           // 第2关历史最少步数
  bestLevel2Time: number;            // 第2关历史最短耗时（秒）

  // 最近一局记录（用于结算界面展示）
  lastRunSteps: number;              // 最近一局步数
  lastRunTime: number;               // 最近一局耗时
  lastRunResult: 'win' | 'lose';    // 最近一局结果
}
```

**写入时机**：
- 胜利结算展示完成后，立即写入 `GameData`
- 若本次步数 < `bestLevel2Steps`，更新 `bestLevel2Steps`
- 若本次耗时 < `bestLevel2Time`，更新 `bestLevel2Time`
- `totalClears` 自增1

**读取时机**：
- 胜利界面展示时，从 `GameData` 读取历史最佳步数和耗时，用于"新纪录"判定（P2功能）

### 8.2 埋点上报方式

- 优先使用微信小游戏内置分析 API `wx.reportEvent`
- 若未接入微信分析，通过自定义 HTTP 上报（批量上报，每30秒或5条合并一次）
- 所有上报失败静默处理，不干扰结算界面展示和操作
- 事件参数中的步数和耗时仅用于统计聚合，不上报用户身份标识

### 8.3 核心指标

| 指标 | 计算方式 | 目标值（参考） |
|------|---------|--------------|
| 第2关通关率 | `lr_victory_show / (lr_victory_show + lr_defeat_show)` | >40% |
| 胜利后重玩率 | `lr_victory_retry / lr_victory_show` | >55% |
| 胜利分享率 | `lr_victory_share / lr_victory_show` | >20% |
| 失败后分享再战率 | `lr_defeat_share_retry / lr_defeat_show` | >30% |
| 失败后流失率 | `lr_defeat_back_home / lr_defeat_show` | <50% |
| 分享成功率 | `lr_share_success / (lr_victory_share + lr_defeat_share_retry)` | >90% |
| 平均通关步数 | 所有胜利局步数的均值 | <25步（参考值，需上线后校正） |
