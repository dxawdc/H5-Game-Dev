# 排行榜系统 系统策划案

**文档版本**：v1.0  **系统ID**：leaderboard  **优先级**：P1  **复杂度**：M
**所属游戏**：围住Boss（Surround the Boss）
**平台**：微信小游戏  **基准分辨率**：648×1152（竖屏）

---

## 一、定位与目标

### 1.1 系统存在的理由

排行榜系统是"围住Boss"社交竞争体系的核心载体，承载玩家之间基于游戏成就的横向比较和排名竞争。在微信小游戏的生态中，好友关系链排行榜是驱动用户回访和社交传播的最强动力之一——玩家不仅想通关，更想在好友圈中证明自己是最优策略的掌握者。

具体而言，本系统解决以下四个核心问题：

1. **社交竞争闭环**：将玩家在关卡2（第2关）中的通关次数、最少步数、最快时间等私人数据转化为可视化的排名，让"通关"这个单一事件升级为"在好友中排名第几"的持续竞争。
2. **重复挑战驱动力**：当玩家看到好友以更少步数或更快时间通关时，会产生"我也可以做到"的挑战欲，推动玩家反复挑战同一关卡以优化成绩，从而延长单次游戏时长和整体留存。
3. **微信生态深度集成**：利用微信开放数据域（Open Data Context）实现好友排行榜的原生体验，玩家无需跳出游戏即可查看好友排名，降低了社交比较的心理门槛。
4. **公平透明的排名体系**：通过"仅记录第2关数据"的统一规则和0.1秒精度的时间计量，确保所有玩家在同一起跑线上竞争，没有任何氪金或等级因素影响排名。

### 1.2 核心目标指标

| 目标维度 | 具体贡献 | 衡量方式 |
|---------|---------|---------|
| 排行榜渗透率 | 至少30%的日活跃用户每天打开一次排行榜 | 埋点 `lb_enter` 次数 / DAU ≥ 0.3 |
| 重复挑战驱动 | 看到好友排名后重新挑战第2关的比例提升 | 排行榜查看后3分钟内进入关卡的比例 ≥ 40% |
| 社交传播 | 排行榜带动分享率 | 排行榜页面"邀请好友"按钮点击率 ≥ 15% |
| 数据上传成功率 | 好友排行榜数据正常展示 | wx.setFriendCloudStorage 调用成功率 ≥ 99% |
| 排行榜打开速度 | 排行榜弹出层从点击到完整展示的时间 | P95 加载耗时 ≤ 1.5秒 |

### 1.3 与其他系统的关系

| 系统ID | 关系类型 | 具体说明 |
|--------|---------|---------|
| `gameplay_core` | 输入 | 对局结束时传入本次通关的步数和耗时数据，用于判定是否刷新最佳成绩 |
| `level_system` | 输入 | 告知当前关卡是否为第2关（仅第2关数据计入排行榜） |
| `level_result` | 依赖 | 结算界面展示本次成绩后，触发排行榜数据上传 |
| `main_menu` | 调用方 | 主界面提供排行榜入口按钮，排行榜以弹出层形式在主界面之上展示 |
| `share_system` | 协同 | 排行榜入口附近提供分享按钮，邀请好友挑战自己的记录 |
| `GameData` | 读写 | 读取 `totalClears`、`bestLevel2Steps`、`bestLevel2Time` 字段，写入最佳成绩 |
| `微信开放数据域` | 基础设施 | 通过 SharedCanvas + wx.setFriendCloudStorage / wx.getFriendCloudStorage 实现好友排名 |

### 1.4 设计红线

- **仅第2关数据原则**：排行榜只记录第2关（9×9网格）的数据。第1关为教学关，不参与排行。任何时候不得将第1关数据同步到开放数据域。
- **最佳成绩覆盖原则**：同一维度下，只有新成绩优于旧成绩时才覆盖上传。例如最少步数当前为20步，新通关用了22步则不触发上传；新通关用了18步则更新上传。减少不必要的开放数据域写入。
- **时间精度统一原则**：所有时间记录精确到 0.1 秒（100毫秒），在UI中统一显示为 "12.3秒" 格式。底层存储为浮点数（单位秒），展示时四舍五入到一位小数。
- **数据自主性原则**：排行榜的数据生命周期完全由玩家自身的游戏行为驱动。不允许运营后台手动修改任何玩家的排行数据。排名变化只能由玩家自己打出更好成绩触发。
- **隐私合规原则**：排行榜仅显示微信好友之间的数据，不展示非好友玩家的排名。不采集、不上传任何用户身份信息（昵称、头像由微信开放数据域自动处理）。

## 二、功能模块拆解

### 2.1 模块总览

| 模块ID | 名称 | 职责描述 | 优先级 | 依赖 |
|--------|------|---------|--------|------|
| `LB-M1` | 三维度排名管理 | 管理三个排名维度（通关数、最少步数、最快时间）的标签/排序/切换逻辑 | P1 | 无 |
| `LB-M2` | 最佳成绩判定器 | 对局结束后判定新成绩是否优于历史最佳，决定是否更新到本地存储和开放数据域 | P1 | `GameData` |
| `LB-M3` | 开放数据域上传器 | 封装 `wx.setFriendCloudStorage` API，将最佳成绩上传到微信好友排行榜存储 | P1 | 微信开放数据域API |
| `LB-M4` | 好友榜单拉取器 | 封装 `wx.getFriendCloudStorage` API，拉取好友排行榜数据并解析为统一格式 | P1 | 微信开放数据域API |
| `LB-M5` | 排行榜UI渲染器 | 在主画布（主域）上渲染排行榜弹出层的全部视觉元素，通过 SharedCanvas 获取开放数据域数据 | P1 | `LB-M4` |
| `LB-M6` | 标签页切换器 | 管理"通关数""最少步数""最快时间"三个标签页的切换状态和数据刷新 | P1 | `LB-M1` |
| `LB-M7` | 自我排名指示器 | 在榜单中高亮当前玩家的条目，显示自己的排名位置 | P1 | `LB-M5` |
| `LB-M8` | 空状态/引导模块 | 排行榜无数据时显示引导文字和"邀请好友"按钮 | P1 | 无 |
| `LB-M9` | 群排行模块（未来） | 支持分享到微信群后查看群内成员的排名（P2功能） | P2 | 微信群排行榜API |

### 2.2 模块详述

#### LB-M1 三维度排名管理

排行榜提供三个维度标签，玩家可横向切换查看不同维度的排名：

| 维度 | 标签文案 | 排序规则 | 数据类型 | 单位 |
|------|---------|---------|---------|------|
| 通关数 | "通关数" | 降序（通关次数越多排名越靠前） | 整数 `int` | 次 |
| 最少步数 | "最少步数" | 升序（步数越少排名越靠前） | 整数 `int` | 步 |
| 最快时间 | "最快时间" | 升序（时间越少排名越靠前） | 浮点数 `float`（精确到0.1秒）| 秒 |

切换维度时，排行榜UI重新排序并刷新列表。切换动画为标签栏下划线滑动过渡（200ms ease-out）。

#### LB-M2 最佳成绩判定器

判定逻辑如下：

```
对局结算（第2关胜利）
       │
       ▼
通关数：totalClears += 1（无论是否刷新纪录，通关次数总是累加）
       │
       ├── 步数判定：newSteps < bestLevel2Steps ?
       │   ├── 是 → 更新 bestLevel2Steps = newSteps，标记 stepsUpdated = true
       │   └── 否 → 不变，stepsUpdated = false
       │
       └── 时间判定：newTime < bestLevel2Time ?
           ├── 是 → 更新 bestLevel2Time = newTime，标记 timeUpdated = true
           └── 否 → 不变，timeUpdated = false
       
       └── 上传触发：若 stepsUpdated || timeUpdated || 需要同步通关数
                      → 调用 LB-M3 上传数据
```

**首次通关特殊处理**：玩家首次通关第2关时，`bestLevel2Steps` 初始为 999，`bestLevel2Time` 初始为 999.0，因此首次通关必定触发更新和上传。

#### LB-M3 开放数据域上传器

上传数据格式（作为 KVDataList 的 value 字符串）：

```json
{
  "totalClears": 15,
  "bestSteps": 18,
  "bestTime": 23.5,
  "updatedAt": "2026-05-09T12:34:56.789Z"
}
```

调用 `wx.setFriendCloudStorage` 上传到微信开放数据域：
- 每次通关后，至少上传一次（含最新通关数）
- 若步数或时间有刷新，一并上传最新最佳数据
- 上传时机：关卡结算界面展示时异步触发，不阻塞UI

#### LB-M4 好友榜单拉取器

- 调用 `wx.getFriendCloudStorage` 拉取所有好友的排行榜数据
- 返回的数据结构包含每个好友的 `openid`、`nickname`、`avatarUrl` 以及自定义的 KVDataList
- 拉取到的数据解析后按当前选中的维度排序
- 缓存策略：拉取到的榜单数据缓存 60 秒，60 秒内重复打开排行榜使用缓存，超过 60 秒重新拉取

#### LB-M5 排行榜UI渲染器

排行榜以弹出层（Popup）形式呈现，在主画布（主域）上渲染。具体布局：

```
┌─────────────────────────────────┐
│  ┌───────────────────────────┐  │
│  │   ✕ 排行榜              │  │  ← 顶部标题栏 + 关闭按钮
│  ├───────────────────────────┤  │
│  │ [通关数] [最少步数] [最快时间] │  │  ← 三个维度标签页
│  ├───────────────────────────┤  │
│  │  🥇 好友A    通关 42次   │  │  ← 排行榜条目（含排名、头像、昵称、数值）
│  │  🥈 好友B    通关 38次   │  │
│  │  🥉 好友C    通关 35次   │  │
│  │  4. 好友D    通关 30次   │  │
│  │  5. 我       通关 15次   │  │  ← 自己的条目高亮显示
│  │  6. 好友E    通关 12次   │  │
│  │  ...                     │  │
│  ├───────────────────────────┤  │
│  │  [📤 邀请好友一起挑战]   │  │  ← 底部邀请按钮
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

排行榜条目设计：
- **前置三名**：前3名显示金银铜奖杯图标（🥇🥈🥉），第4名起显示数字
- **当前玩家**：自己的条目使用特殊背景色或边框高亮（如浅金色边框），昵称后标注"我"
- **头像&昵称**：通过 SharedCanvas 从开放数据域获取好友头像和昵称，在开放数据域侧渲染
- **数值展示**：不同维度显示不同单位（次/步/秒），高亮当前玩家的数值

由于微信小游戏的开放数据域限制，排行榜UI的渲染架构采用**双域协作**方式：
- **主域（Main Context）**：渲染排行榜的框架元素（背景面板、标题、标签栏、关闭按钮、底部按钮等）以及排名数字和奖杯图标
- **开放数据域（Open Data Context）**：渲染每个好友的头像、昵称和数值，通过 `sharedCanvas` 传递到主域

#### LB-M6 标签页切换器

- 三个标签横向排列，当前选中标签有下划线指示
- 点击标签切换维度时，清空当前列表，显示加载中动画，然后根据新维度重新排序展示
- 如果该维度下无数据，显示空状态提示

#### LB-M7 自我排名指示器

- 在好友榜单中定位当前玩家的位置
- 若当前玩家不在好友榜单的前100名中，显示"击败了XX%的好友"的鼓励文案
- 排名计算方式：`击败百分比 = (1 - 当前排名 / 好友总数) × 100%`

#### LB-M8 空状态/引导模块

- 当开放数据域返回空数据时（玩家或好友均未上传过成绩），展示引导界面
- 引导文案："还没有排行数据，开始挑战第2关吧！"
- 显示"开始挑战"快捷按钮，点击直接跳转到关卡2
- 显示"邀请好友"按钮，点击触发 `wx.shareAppMessage`

#### LB-M9 群排行模块（未来）

P2功能规划：
- 玩家可将自己的成绩分享到微信群
- 分享卡片携带群ID和成绩数据
- 群内其他玩家点击分享卡片进入游戏时，自动拉取同群成员的排行榜
- 使用微信的群排行榜相关API（需微信小游戏支持）
- 群排行与好友排行互不干扰，通过独立标签页切换

## 三、核心玩法规则

### 3.1 排行榜数据采集规则

| 规则编号 | 规则内容 | 说明 |
|---------|---------|------|
| R1 | 仅第2关（9×9网格）数据计入排行榜 | 第1关为教学关，任何成绩均不上传 |
| R2 | 仅胜利对局的数据计入排行榜 | 失败/放弃/步数耗尽的对局不计入任何维度 |
| R3 | 成绩必须优于历史最佳才覆盖上传 | 步数更低或时间更快才触发更新，降低无效写入 |
| R4 | 通关数始终递增 | 每次胜利 totalClears += 1，通关数只增不减 |
| R5 | 首次通关必定触发上传 | `bestLevel2Steps` 和 `bestLevel2Time` 的初始值为极大值 |
| R6 | 排行榜上传最多延迟 3 秒 | 结算后 3 秒内必须发起上传，超时则强制上传 |

### 3.2 排名计算规则

| 维度 | 排名方向 | 同分处理 | 说明 |
|------|---------|---------|------|
| 通关数 | 降序（高→低） | 通关数相同则按更新时间升序，先达到该通关数的排名靠前 | 鼓励持续挑战 |
| 最少步数 | 升序（低→高） | 步数相同则按时间升序，用时更短的排名靠前 | 步数优先，时间次之 |
| 最快时间 | 升序（低→高） | 时间相同则按步数升序，步数更少的排名靠前 | 时间优先，步数次之 |

### 3.3 时间精度与格式规则

- **内部存储**：使用 `number` 类型，单位为秒，精确到小数点后 1 位（0.1秒精度）
- **存储示例**：23.5（表示23.5秒）、12.0（表示12.0秒）
- **UI展示**：固定一位小数格式 "XX.X秒"，如 "23.5秒"、"12.0秒"
- **四舍五入**：上传前四舍五入保留一位小数，比较时使用四舍五入后的值
- **边界处理**：通关时间 ≥ 999.9 秒时视为异常数据，不上传（建议设置最大有效时间 300 秒）

### 3.4 数据上传频率限制

| 限制项 | 限制值 | 说明 |
|-------|-------|------|
| 单次通关触发上传次数 | 最多 1 次 | 单次胜利结算只触发一次 `wx.setFriendCloudStorage` |
| 每日上传频率 | 不限制 | 微信开放数据域无频率限制，但建议每次通关最多触发一次 |
| KVDataList 大小 | 不超过 2KB | 上传的 JSON 字符串需精简字段名 |
| 上传重试 | 最多 2 次 | 上传失败后间隔 1 秒重试，2 次均失败则放弃，不影响用户 |

### 3.5 进入/退出条件

**进入条件**：
- 玩家在主界面点击排行榜入口按钮（奖杯图标）
- 触发 `leaderboardOpened` 事件
- 排行榜弹出层从右侧滑入
- 异步拉取好友榜单数据

**退出条件**：
- 玩家点击关闭按钮（×）
- 玩家点击遮罩层区域
- 触发 `leaderboardClosed` 事件
- 排行榜弹出层向右滑出（200ms ease-in）

## 四、完整流程与状态机

### 4.1 排行榜弹出层状态机

```
                ┌───────────────────┐
                │     CLOSED        │ ← 初始状态
                └────────┬──────────┘
                         │ 点击排行榜入口按钮
                         ▼
                ┌───────────────────┐
                │  OPENING          │ ← 滑入动画（200ms）
                └────────┬──────────┘
                         │ 动画完成
                         ▼
                ┌───────────────────┐
         ┌─────│  LOADING           │ ← 开始拉取好友数据
         │     └────────┬──────────┘
         │              │ 数据拉取完成
         │              ▼
         │     ┌───────────────────┐
         │     │  DISPLAYING       │ ← 榜单展示，可交互
         │     └──┬────┬────┬──────┘
         │        │    │    │
         │        │    │    └──────────┐
         │        │    │               │
         │        │    │    点击标签切换维度
         │        │    │               │
         │        │    │               ▼
         │        │    │      ┌────────────────┐
         │        │    └──────│  SWITCHING     │ ← 切换动画（200ms）
         │        │           └────────┬───────┘
         │        │                    │ 排序完成
         │        │                    ▼
         │        │           ┌────────────────┐
         │        └──────────►│  DISPLAYING    │ ← 新维度榜单展示
         │                    └────────────────┘
         │
         │  点击关闭/遮罩
         ▼
  ┌──────────────┐
  │  CLOSING     │ ← 滑出动画（200ms）
  └──────┬───────┘
         │ 动画完成
         ▼
  ┌──────────────┐
  │  CLOSED      │ ← 回到初始状态
  └──────────────┘
```

### 4.2 数据上传流程（通关结算时）

```
[玩家胜利通关第2关]
       │
       ▼
[LevelResult 结算界面展示]
       │
       ▼
[GameData 更新] → totalClears += 1
       │
       ▼
[最佳成绩判定] → newSteps < bestLevel2Steps? → 是 → 更新
                → newTime < bestLevel2Time?   → 是 → 更新
       │
       ▼
[触发上传] → 构建 KVData（含 totalClears、bestSteps、bestTime、updatedAt）
       │
       ▼
[wx.setFriendCloudStorage] → 异步调用
       │
       ├── 成功 → 记录上传时间，无用户反馈
       │
       └── 失败（重试最多2次）→ 均失败 → 静默放弃，不影响用户
```

### 4.3 数据拉取流程（打开排行榜时）

```
[玩家点击排行榜入口]
       │
       ▼
[检查缓存] → 距离上次拉取 < 60秒?
       │
       ├── 是 → 使用缓存数据直接渲染
       │
       └── 否 → 调用 wx.getFriendCloudStorage
                 │
                 ▼
           [解析返回数据] → 遍历好友列表，解析 KVDataList 中的 JSON
                 │
                 ├── 成功 → 按当前维度排序 → 更新缓存时间戳 → 渲染
                 │
                 └── 失败 → 使用缓存数据（如有）或显示空状态
```

### 4.4 异常流程

1. **开放数据域API调用失败**：
   - 上传失败（`wx.setFriendCloudStorage` 返回错误）：重试2次，均失败则静默放弃，不弹出任何错误提示
   - 拉取失败（`wx.getFriendCloudStorage` 返回错误）：使用缓存数据展示，若缓存也为空则显示空状态引导

2. **KVData 数据格式解析异常**：
   - 某个好友的 value JSON 解析失败 → 跳过该好友，不影响其他好友的展示
   - 数据字段缺失（如 missing `bestSteps`）→ 该维度下该好友显示 "-"

3. **SharedCanvas 渲染失败**：
   - 开放数据域无法正确渲染头像/昵称 → 降级显示占位头像（灰色圆形）和 "未知好友"
   - 主域框架仍正常展示，不影响用户关闭排行榜

4. **玩家数据从未上传**：
   - 首次打开排行榜时，自己的数据不在好友榜单中
   - 在榜单末尾显示自己的条目，数值显示 "--"

5. **好友数量为0**：
   - 玩家的微信好友中无人玩过本游戏
   - 排行榜显示空状态："还没有好友成绩，邀请好友一起挑战吧！"

## 五、边界条件与异常处理

### 5.1 数值边界

| 边界场景 | 预期行为 | 防御措施 |
|---------|---------|---------|
| 步数为0通关（理论上不可能）| 不上传，视为数据异常 | 判定器校验 steps > 0 才允许更新 |
| 通关时间为0或负数 | 不上传，视为数据异常 | 判定器校验 time > 0 才允许更新 |
| 通关时间超过 300 秒 | 不上传，视为数据异常 | 判定器校验 time <= 300 才允许更新 |
| 通关数超过 9999 | 正常展示，不做截断 | 微信开放数据域无数值上限 |
| 好友数量超过 100 | 仅展示前 100 名好友 | 榜单列表做截断处理，显示"仅展示前100名好友"提示 |
| 同一好友上传多条数据 | 以最新的 KVData 为准 | 微信开放数据域按 openid 去重 |

### 5.2 非法操作拦截

| 非法操作 | 拦截层级 | 反馈方式 |
|---------|---------|---------|
| 在排行榜LOADING状态重复点击维度标签 | UI层 | 标签按钮置灰，禁止重复点击 |
| 在开放数据域加载完成前点击关闭后立即打开 | 逻辑层 | 关闭动画进行中禁止再次打开 |
| 结算界面展示完成前重复上传 | 逻辑层 | 上传锁（flag `isUploading`），上传完成前忽略后续请求 |
| 上传数据中的 `updatedAt` 早于本地记录 | 数据层 | 拒绝以旧数据覆盖新数据 |

### 5.3 幂等设计

| 操作类型 | 幂等key | 实现方式 |
|---------|--------|---------|
| `wx.setFriendCloudStorage` | `updatedAt` 时间戳 | 每次上传使用当前时间戳，微信开放数据域以最新写入为准 |
| 关卡结算数据更新 | `roundId` | 同一 roundId 只触发一次数据更新和上传 |
| 排行榜拉取 | 无（可重复拉取） | 60 秒缓存去重 |

### 5.4 微信开放数据域特有边界

| 边界场景 | 预期行为 |
|---------|---------|
| 用户未授权微信好友关系 | 排行榜显示"需要好友关系授权"提示，引导用户在微信设置中开启 |
| 微信小游戏未开通开放数据域 | 排行榜显示功能不可用提示，引导联系客服 |
| 开放数据域初始化失败 | 排行榜弹出框架但内容区显示"数据加载失败" |
| SharedCanvas 尺寸不一致 | 主域和开放数据域的坐标系需要对齐，确保渲染位置正确 |
| 微信版本过低不支持开放数据域 | 检测 API 是否存在（`typeof wx.setFriendCloudStorage === 'undefined'`），显示提示 |

## 六、前后端通信设计

### 6.1 通信方式

排行榜系统主要依赖微信小游戏开放数据域API，无自建后端服务器。所有通信通过微信客户端与微信服务器之间的原生API完成。

| API名称 | 方向 | 用途 | 调用时机 |
|--------|------|------|---------|
| `wx.setFriendCloudStorage` | 客户端→微信服务器 | 上传玩家最佳成绩到好友排行榜存储 | 第2关胜利结算时 |
| `wx.getFriendCloudStorage` | 客户端→微信服务器 | 拉取好友的排行榜数据 | 打开排行榜弹出层时 |
| `wx.shareAppMessage` | 客户端→微信客户端 | 分享邀请到好友/群 | 点击"邀请好友"按钮时 |

主域与开放数据域之间的通信通过 SharedCanvas + `wx.postMessage` 实现：

```
主域（Main Context）                   开放数据域（Open Data Context）
     │                                          │
     │── postMessage({type:'switchTab',dim})───►│  切换维度标签
     │                                          │── 重新排序并渲染到 SharedCanvas
     │◄── SharedCanvas 更新 ──────────────────│
     │                                          │
     │── postMessage({type:'refresh'})─────────►│  手动刷新
     │                                          │── 重新拉取并渲染
     │◄── SharedCanvas 更新 ──────────────────│
```

### 6.2 关键事件/信号

所有事件使用 `camelCase` 命名，通过全局事件总线派发。

| 事件名 | 方向 | 载荷 | 触发时机 |
|--------|------|------|---------|
| `leaderboardOpened` | UI→排行榜 | `{ defaultTab: string }` | 排行榜弹出层开始滑入 |
| `leaderboardClosed` | 排行榜→UI | `{ duration: number, lastTab: string }` | 排行榜弹出层完全关闭 |
| `rankingUpdated` | LevelResult→排行榜 | `{ totalClears: number, bestSteps: number, bestTime: number }` | 通关结算后触发 |
| `leaderboardDataLoaded` | 开放数据域→主域 | `{ tab: string, entries: RankEntry[], selfRank: number }` | 拉取并解析完成 |
| `leaderboardDataErrored` | 开放数据域→主域 | `{ errorCode: number, errorMsg: string }` | 拉取出错时 |
| `leaderboardTabSwitched` | UI→开放数据域 | `{ from: string, to: string }` | 点击切换维度标签时 |
| `scoreUploaded` | 排行榜→全局 | `{ success: boolean, dimension: string }` | 最佳成绩上传完成 |
| `rankShareTriggered` | 排行榜→分享系统 | `{ shareType: 'invite_friend' \| 'challenge' }` | 点击邀请好友按钮时 |
| `friendInvited` | 分享系统→排行榜 | `{ shareSuccess: boolean }` | 分享邀请完成回调 |
| `leaderboardCacheHit` | 排行榜→分析 | `{ cacheAge: number }` | 使用缓存数据展示时 |

### 6.3 开放数据域数据格式

**上传数据结构（KVDataList 中的 value 字符串）**：

```typescript
// 上传到 wx.setFriendCloudStorage 的 KVData 格式
interface UploadData {
  totalClears: number;       // 累计通关数，如 15
  bestSteps: number;         // 历史最少步数，如 18
  bestTime: number;          // 历史最快时间（秒），如 23.5
  updatedAt: string;         // ISO 8601 时间戳，如 "2026-05-09T12:34:56.789Z"
}

// 单个 KVData 项
// key: "score_data"
// value: JSON.stringify(uploadData)
```

**拉取数据结构（wx.getFriendCloudStorage 返回）**：

```typescript
interface RawFriendData {
  openid: string;            // 好友的 openid
  nickname: string;          // 微信昵称
  avatarUrl: string;         // 微信头像 URL
  KVDataList: Array<{
    key: string;             // "score_data"
    value: string;           // JSON 字符串
  }>;
}

// 解析后的排行榜条目
interface RankEntry {
  openid: string;
  nickname: string;
  avatarUrl: string;
  totalClears: number;
  bestSteps: number;
  bestTime: number;
  updatedAt: string;
  rank: number;              // 当前维度下的排名
}
```

### 6.4 错误码

| 错误码 | 名称 | 含义 | 处理方式 |
|--------|------|------|---------|
| -1 | `DATA_PARSE_ERROR` | 拉取数据 JSON 解析失败 | 跳过该条数据，继续处理其他 |
| -2 | `UPLOAD_FAILED` | 上传到开放数据域失败 | 重试2次，均失败则静默放弃 |
| -3 | `FETCH_FAILED` | 拉取好友数据失败 | 使用缓存数据，缓存为空时显示空状态 |
| -4 | `INVALID_STEPS` | 步数数据异常（≤0）| 不更新，不触发上传 |
| -5 | `INVALID_TIME` | 时间数据异常（≤0 或 >300）| 不更新，不触发上传 |
| -6 | `OPEN_DATA_UNAVAILABLE` | 开放数据域API不可用 | 显示功能不可用提示 |
| -7 | `SHARED_CANVAS_ERROR` | SharedCanvas 渲染异常 | 降级显示文字信息 |

| 成就徽章展示 | 在排行榜条目上展示玩家获得的成就徽章 | P2 |
| 排行榜主题皮肤 | 节假日/活动期间更换排行榜面板的视觉风格 | P2 |
| 排行榜通知 | 好友刷新纪录时发送服务通知（需微信模板消息）| P3 |

### 9.5 明确不做

| 事项 | 不做原因 | 备注 |
|------|---------|------|
| 第1关数据计入排行榜 | 第1关为教学关，数据不具备竞争可比性 | 严格锁定仅第2关 |
| 道具使用次数排名 | 道具设计是为了辅助通关，不应成为竞争维度 | 保持排名的纯粹性 |
| 实时在线排行榜 | 微信小游戏的开放数据域仅支持异步数据，不支持实时WebSocket | 技术限制，非实时 |
| 排行榜防沉迷限制 | 微信小游戏的防沉迷系统由微信SDK统一管理 | 无需自行实现 |
| 与排行榜挂钩的付费特权 | 任何形式的付费加速/保排名功能严重破坏公平性 | 核心价值观底线 |
| 显示非好友的全服排行 | 涉及用户隐私和数据合规风险 | 如业务需要建议另做服务端方案 |
