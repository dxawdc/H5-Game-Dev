# 主界面系统 系统策划案

**文档版本**：v1.0  **系统ID**：main_menu  **优先级**：P0  **复杂度**：S
**所属游戏**：围住Boss（Surround the Boss）
**平台**：微信小游戏  **基准分辨率**：648×1152（竖屏）

---

## 一、定位与目标

### 1.1 系统定位

主界面系统是玩家进入游戏后看到的第一个交互界面（加载流程完成后），是游戏的大厅和功能中枢。所有核心功能的入口均汇集于此，玩家通过主界面导航到各个子系统（关卡、排行榜、设置等）。主界面承担着"第一印象"的重要角色，其视觉品质和交互流畅度直接影响玩家的留存意愿。

### 1.2 核心目标

1. **品牌传达**：通过游戏Logo、背景美术、动态装饰元素，在第一时间向玩家传递"围住Boss"的卡通休闲风格和核心玩法暗示
2. **功能导航**：提供清晰直观的功能入口，让玩家零学习成本即可找到"开始挑战"按钮进入核心玩法
3. **社交传播**：通过分享按钮和排行榜入口，驱动微信小游戏的社交裂变
4. **商业转化**：为广告位、道具购买、换装等付费/观看广告入口提供展示位置（P1/P2逐步接入）
5. **留存驱动**：通过每日分享奖励、排行榜竞争、活动入口等方式提升玩家回访率

### 1.3 与上下游系统的关系

| 上游系统 | 交互方式 | 说明 |
|---------|---------|------|
| Loading系统 | 场景切换 | Loading加载完成后自动跳转至主界面 |
| 关卡结算系统 | 场景切换 | Level 2通关后自动返回主界面 |

| 下游系统 | 交互方式 | 说明 |
|---------|---------|------|
| 核心玩法系统 | 场景切换 | 点击"开始挑战"进入第1关 |
| 关卡系统 | 读取关卡配置 | 根据当前关卡状态决定进入第1关还是第2关 |
| 排行榜系统 | 弹出层（Popup） | 点击排行榜按钮打开排行榜弹出层 |
| 设置系统 | 弹出层（Popup） | 点击设置按钮打开设置弹出层 |
| 分享系统 | 微信原生API调用 | 点击分享按钮触发 wx.shareAppMessage |
| 道具系统 | 读取/写入道具数据 | 每日首次分享获得随机道具奖励 |
| 广告系统 | 弹出层/卡片展示 | 主界面底部/侧边广告位（P1） |

### 1.4 玩家画像与使用场景

- **新玩家**：首次进入，看到Logo和"开始挑战"按钮，点击直接进入第1关教学关
- **老玩家**：已通关第1关未通关第2关，进入时看到"继续挑战"，点击进入第2关
- **通关玩家**：已通关第2关，进入时看到"开始挑战"，点击再次进入第2关
- **社交型玩家**：每日首次进入先点击分享获取道具奖励，再查看排行榜好友排名，最后开始挑战
- **活跃型玩家**：每日完成挑战后，查看排行榜、设置调整、参与活动（P2）

### 2.2 P0模块详述（M1-M3）

#### M1 背景展示
- **视觉**：全屏背景图，648×1152分辨率，休闲卡通风格，浅色渐变为主
- **动态元素**：背景中包含轻微动的装饰元素（飘浮的云朵、闪烁的星星、微风中摆动的草木等），营造生动氛围
- **适配策略**：采用 `cover` 策略缩放，确保不同屏幕比例下背景完整填充
- **安全区**：四边各保留40px安全区，避免异形屏/刘海屏裁剪关键内容

#### M2 Logo展示
- **位置**：背景中上方，距顶部约160px，居中布局
- **内容**：游戏标题"围住Boss"，卡通风格字体，带轻微阴影或描边增强可读性
- **尺寸**：约为屏幕宽度的60%（约390px），高度约120px
- **动画**：进入主界面时Logo可做淡入/弹跳等入场动画（500ms），日常展示可带微弱呼吸光效

#### M3 开始挑战按钮
- **位置**：屏幕中下方，距底部约350px，居中布局
- **视觉**：圆角矩形按钮，渐变色（暖色系，橙→红渐变），白色粗体文字
- **尺寸**：300×80px（符合88px最小触控高度）
- **文案逻辑**：
  - 玩家未通关第1关（首次进入）→ 显示"开始挑战"
  - 玩家已通关第1关但未通关第2关 → 显示"继续挑战"
  - 玩家已通关第2关 → 显示"再来一局"
- **交互**：点击后按钮缩小再恢复（缩放动画 0.95→1.0，100ms），随后场景切换到关卡界面
- **文字下标**：按钮下方显示小号提示文字，如"第1关·7×7教学"或"第2关·9×9挑战"

### 2.3 P1模块详述（M4-M8）

#### M4 排行榜入口
- **位置**：屏幕右侧中部，与设置对称分布
- **视觉**：圆形图标按钮，直径64px，奖杯图标
- **点击区域**：扩展至88×88px（保证触控）
- **交互**：点击后从右侧滑入排行榜弹出层（半透明遮罩+排行榜面板），排行榜面板宽度约500px，高度约700px
- **数据**：调用微信排行榜API（wx.getFriendCloudStorage），显示好友按通关次数/最少步数/最快时间排名

#### M5 设置入口
- **位置**：屏幕左侧中部，与排行榜对称分布
- **视觉**：圆形图标按钮，直径64px，齿轮图标
- **交互**：点击后从左侧滑入设置弹出层，包含：音效开关（滑块/切换按钮）、震动开关、游戏版本号显示

#### M6 分享入口
- **位置**：屏幕右上角，距安全区边缘约20px
- **视觉**：分享图标（箭头），带每日首次分享红点提示
- **交互**：点击后调用 wx.shareAppMessage，分享卡片图片+文案："来围住这个Boss！你能用几步困住它？"
- **每日首次分享奖励**：分享成功后检测 `dailyShareCount`，若为0则给予随机道具奖励，弹出道具获得弹窗，并将 `dailyShareCount` 置为1

#### M7 道具展示
- **位置**：屏幕底部，开始挑战按钮下方
- **内容**：横向排列四个道具图标（多走1步、返回上1步、重置本关、提示），每个图标64×64px，下方显示持有数量
- **空态**：数量为0时图标置灰，右上角显示"+"号引导获取
- **交互**：点击道具图标→弹出道具说明浮窗；点击"+"号→弹出获取方式选择（分享/看广告）

#### M8 每日分享奖励提示
- **触发条件**：每日首次分享成功后
- **表现形式**：从屏幕中央弹出道具获得弹窗，显示获得的道具图标和数量，带"恭喜获得"标题和关闭按钮
- **动画**：弹窗带缩放弹入效果（0.5→1.0，300ms ease-out）
- **特殊case**：背包已满时（各道具数量达上限），弹窗提示"道具已达上限"，不增加数量

## 三、核心玩法规则（UI导航规则）

### 3.1 主界面按钮文案状态机

根据玩家进度，主界面核心按钮文案和下一跳目标动态变化：

| 玩家状态 | 判定条件 | 按钮文案 | 目标关卡 | 下标提示 |
|---------|---------|---------|---------|---------|
| 新手 | `totalClears == 0` 且从未通关关卡1 | 开始挑战 | 第1关（7×7） | "第1关·教学" |
| 卡关1 | 关卡1失败次数≥3且从未通关 | 开始挑战 | 第1关（7×7） | "再试一次" |
| 通关1未通关2 | 通关关卡1记录存在且 `totalClears == 0` | 继续挑战 | 第2关（9×9） | "第2关·挑战" |
| 通关2 | `totalClears > 0` | 再来一局 | 第2关（9×9） | "最佳{bestLevel2Steps}步" |

### 3.2 按钮层级与点击区域规范

所有主界面按钮遵循以下规范：

- 最小触控区域：88×88px（逻辑像素）
- 按钮间最小间距：20px
- 视觉反馈：按下状态（scale 0.95 + 颜色变暗 20%）
- 点击防连点：点击后按钮立即禁用（disabled状态），待场景切换完成后恢复

### 3.3 弹出层导航规则

排行榜、设置等次级界面以弹出层（Popup）形式展示：

- **遮罩层**：半透明黑色（rgba 0,0,0,0.5），点击遮罩区域可关闭弹出层
- **关闭按钮**：弹出层右上角提供关闭按钮（×），直径至少44×44px
- **层叠规则**：同一时间仅显示一个弹出层；打开新弹出层时先关闭当前弹出层
- **动画**：弹出层从对应方向滑入（排行榜从右，设置从左），持续200ms ease-out

## 四、完整流程与状态机

### 4.1 主界面状态定义

| 状态 | 说明 |
|-----|------|
| `IDLE` | 主界面空闲态，展示所有功能入口，等待用户操作 |
| `TRANSITIONING_IN` | 从Loading场景切换到主界面的过渡动画中 |
| `POPUP_ACTIVE` | 有弹出层（排行榜/设置）处于打开状态 |
| `SHARING` | 正在调用微信分享API |
| `NAVIGATING` | 正在切换到其他场景（关卡/结算等） |

### 4.3 详细流程描述

#### 流程A：首次进入主界面

1. Loading系统完成资源加载，触发 `mainMenuEntered` 事件
2. 主界面进入 `TRANSITIONING_IN` 状态
3. 背景渲染 → Logo淡入（0→1 opacity，500ms）→ 按钮和图标依次淡入（各间隔100ms）
4. 读取 `GameData` 判断玩家状态，设置按钮文案
5. 检测 `dailyShareCount` 是否为0，若是则在分享按钮上显示红点
6. 进入 `IDLE` 状态，所有按钮恢复交互

#### 流程B：开始挑战

1. 玩家点击"开始挑战/继续挑战/再来一局"按钮
2. 按钮禁用，触发 `challengeStarted` 事件
3. 进入 `NAVIGATING` 状态
4. 播放按钮点击动画（100ms）
5. 根据当前玩家状态确定目标关卡ID：
   - 未通关第1关 → 关卡ID = 1
   - 已通关第1关未通关第2关或已通关第2关 → 关卡ID = 2
6. 场景切换到关卡界面（`GameLevel` 场景），传入关卡ID

#### 流程C：返回主界面（关卡2通关后）

1. 关卡结算系统展示胜利界面
2. 玩家点击"返回主页"按钮（或自动倒计时结束后）
3. 场景切换回主界面
4. 主界面重新读取玩家数据，更新按钮文案和道具展示
5. 进入 `IDLE` 状态

#### 流程D：每日首次分享

1. 玩家点击分享按钮
2. 进入 `SHARING` 状态
3. 调用 `wx.shareAppMessage`，传入分享配置（标题、图片、路径）
4. 分享完成回调（无论用户实际是否发送成功，微信均触发回调）：
   - 读取 `GameData.dailyShareCount`
   - 若 `dailyShareCount == 0`：
     - 从4种道具（extra_step, undo, reset, hint）中随机选取1种
     - 道具数量 +1
     - `dailyShareCount` 设为1
     - 保存到 `localStorage`
     - 弹出道具获得弹窗
   - 若 `dailyShareCount > 0`：无操作
5. 分享按钮红点消失
6. 回到 `IDLE` 状态

### 4.4 关卡间自动切换流程（特殊流程）

此流程虽然是关卡系统主导，但主界面需要感知其状态以正确展示入口文案：

```
[主界面] → 开始挑战 → [关卡1]
                         │ 通关
                         ▼
                     [关卡2]（自动进入，无主界面中转）
                         │
                    ┌────┴────┐
                    │         │
                 通关      失败
                    │         │
                    ▼         ▼
               [结算-胜]   [结算-败]
                    │         │
                    ▼         ▼
               [主界面]    [关卡2重试/主界面]
```

关键规则：
- 关卡1通关后**自动进入**关卡2，不经过主界面
- 关卡2通关后返回主界面，主界面显示"再来一局"文案
- 关卡2失败后，玩家可选择重试关卡2或返回主界面

## 五、边界条件与异常处理

### 5.1 网络相关

| 场景 | 预期行为 |
|-----|---------|
| 无网络时点击排行榜 | 弹出提示"网络异常，请检查网络连接"，排行榜展示空状态或本地缓存数据 |
| 无网络时点击分享 | 微信分享API调用失败，弹出提示"分享失败，请检查网络"，不触发奖励逻辑 |
| 分享API调用超时 | 设定5秒超时，超时后提示"分享超时，请重试"，不触发奖励逻辑 |
| 网络从断开到恢复 | 主界面不自动刷新，需要用户主动操作（点击排行榜等触发重新请求） |

### 5.2 数据异常

| 场景 | 预期行为 |
|-----|---------|
| localStorage数据损坏 | 读取 `GameData` 时捕获解析异常，重置为默认值，静默修复 |
| 关卡配置缺失 | 开始挑战时检测关卡配置是否存在，若缺失则弹出提示"关卡配置加载失败"并返回主界面 |
| 道具配置缺失 | 道具图标展示区域显示占位图标，控制台输出警告 |
| 玩家数据字段缺失 | 使用 `||` 默认值兜底（如 `dailyShareCount = data.dailyShareCount || 0`） |

### 5.3 UI/交互异常

| 场景 | 预期行为 |
|-----|---------|
| 快速连点开始挑战按钮 | 按钮点击后立即禁用（disabled），防止多次场景切换 |
| 极窄屏幕（<375px逻辑宽度） | 按钮和图标自动缩小适配，保持布局不溢出 |
| 异形屏/刘海屏 | 安全区40px自动适配，关键内容不进入安全区 |
| 低端机型动画卡顿 | 检测帧率低于30fps时自动降低动画复杂度（跳过入场动画、减少粒子效果） |
| 排行榜数据为空 | 显示空状态提示："暂无好友排行，邀请好友一起玩吧！" |
| 本地存储空间不足 | 保存数据时捕获 `QuotaExceededError`，提示"存储空间不足，请清理后重试" |

### 5.4 并发与时序异常

| 场景 | 预期行为 |
|-----|---------|
| 正在分享时点击其他按钮 | 分享状态下屏蔽所有非必要操作，分享完成后恢复 |
| 弹出层未关闭时触发场景切换 | 先关闭弹出层再执行场景切换 |
| 频繁切场景（内存不足） | 场景切换前检查可用内存，低于阈值时触发资源清理（卸载不使用的纹理） |

## 六、前后端通信设计

### 6.1 通信方式

微信小游戏采用"单页面 + 场景管理"的纯前端架构，无后端服务器。所有数据存储在微信小游戏的 `wx.setStorageSync` / `wx.getStorageSync` 或前端 `localStorage` 中。

排行榜数据通过微信提供的开放数据域（Open Data Context / `wx.setFriendCloudStorage`）实现，主界面通过共享画布（SharedCanvas）渲染。

| 数据类型 | 存储方式 | 读写时机 |
|---------|---------|---------|
| 玩家进度（通关数、最佳步数） | localStorage | 主界面初始化时读取；关卡结算时写入 |
| 道具库存 | localStorage | 主界面初始化时读取；分享/广告获得道具时写入 |
| 每日次数（分享、广告） | localStorage + 时间戳 | 每次分享/广告后写入，初始化时检查日期是否变更 |
| 排行榜数据 | 微信开放数据域 | 打开排行榜时读取 |
| 分享配置 | 静态数据（代码内定义） | 点击分享时读取 |

### 6.2 事件总线协议

主界面通过全局事件总线（EventEmitter）与上下游系统通信，事件全部采用 camelCase 命名：

| 事件名 | 方向 | 载荷 | 触发时机 |
|--------|------|------|---------|
| `mainMenuEntered` | 主界面 → 全局 | `{}` | 主界面场景加载并完成入场动画后 |
| `challengeStarted` | 主界面 → 关卡系统 | `{ levelId: number }` | 玩家点击开始挑战按钮时 |
| `shareTriggered` | 主界面 → 分享系统 | `{ shareType: 'daily' }` | 玩家点击分享按钮时 |
| `shareCompleted` | 分享系统 → 主界面 | `{ isFirstToday: boolean, rewardItem?: ItemType }` | 分享API回调完成时 |
| `leaderboardOpened` | 主界面 → 排行榜 | `{}` | 排行榜弹出层打开时 |
| `leaderboardClosed` | 排行榜 → 主界面 | `{}` | 排行榜弹出层关闭时 |
| `settingsOpened` | 主界面 → 设置系统 | `{}` | 设置弹出层打开时 |
| `settingsClosed` | 设置系统 → 主界面 | `{ soundEnabled: boolean, vibrationEnabled: boolean }` | 设置弹出层关闭时 |
| `sceneTransitionStarted` | 场景管理器 → 全局 | `{ from: string, to: string }` | 场景切换开始时 |
| `sceneTransitionCompleted` | 场景管理器 → 全局 | `{ scene: string }` | 场景切换完成时 |
| `dailyRewardClaimed` | 主界面 → 道具系统 | `{ itemType: string, quantity: number }` | 每日分享奖励发放时 |

### 6.3 数据存储设计

本地存储键值规划（全部挂载在 `GameData` 单例下）：

```typescript
// 主界面相关存储字段
interface MainMenuStorageData {
  // 每日分享标记（含日期，用于判断是否跨日重置）
  lastShareDate: string;         // ISO日期字符串 "2026-05-09"
  dailyShareCount: number;       // 0 或 1

  // 玩家进度（由关卡系统/结算系统写入）
  level1Cleared: boolean;        // 第1关是否通关
  totalClears: number;           // 第2关累计通关次数
  bestLevel2Steps: number;       // 第2关最少步数
  bestLevel2Time: number;        // 第2关最快时间（秒）

  // 道具库存
  items: Record<string, number>; // 如 { extra_step: 2, undo: 1, reset: 0, hint: 3 }

  // 设置
  soundEnabled: boolean;
  vibrationEnabled: boolean;
}
```

## 八、数据埋点需求

### 8.1 埋点总表

以下为推荐埋点事件，用于后续运营分析和产品迭代决策。

| 事件ID | 事件名 | 触发时机 | 上报字段 | 用途 |

### 8.2 埋点上报方式

- 使用微信小游戏 `wx.reportEvent`（若接入微信数据分析）或自定义上报
- 每事件延迟上报，非关键路径事件可合并批量上报（每30秒或临界事件触发时）
- 上报失败静默处理，不影响用户体验

### 8.3 核心指标

| 指标 | 计算方式 | 目标值（参考） |
|-----|---------|--------------|
| 开始挑战按钮点击率 | 点击次数 / 主界面PV | >60% |
| 分享入口点击率 | 分享点击次数 / 主界面PV | >25% |
| 每日首次分享转化率 | 每日首次分享人数 / DAU | >40% |

| 主界面→关卡的流失率 | 进入主界面但未点击开始的用户占比 | <10% |

## 九、迭代规划

### 9.1 版本路线图

| 版本 | 阶段 | 包含模块 | 预计人力 | 里程碑 |
|-----|------|---------|---------|--------|
| v1.0.0 | MVP核心 | M1背景、M2 Logo、M3开始挑战 | 1前端+1美术（3天） | 可进入关卡的核心闭环 |
| v1.1.0 | 社交基础 | M4排行榜、M6分享、M8分享奖励 | 1前端+0.5美术（2天） | 社交裂变能力就绪 |
| v1.2.0 | 用户体验 | M5设置、M7道具展示 | 1前端+0.5美术（2天） | 基础体验完整 |
| v2.0.0 | 商业化 | M9换装入口、M13广告位 | 1前端+1美术（3天） | 商业化能力就绪 |
| v2.1.0 | 社交深化 | M10更多游戏、M12好友圈 | 1前端（2天） | 用户留存提升 |
| v2.2.0 | 运营能力 | M11活动入口、节日限定背景 | 1前端+1美术（2天） | 运营活动支撑 |

### 9.2 P0-P2模块优先级依据

- **P0（MVP）**：背景、Logo、开始挑战按钮是主界面存在的核心意义，缺少任何一个都无法构成"主界面"
- **P1（核心体验）**：排行榜、分享、设置、道具展示是提升用户粘性和社交传播的关键，离开它们游戏依然可玩但体验不完整
- **P2（长期运营）**：换装、更多游戏、活动、好友圈、广告位等属于增值功能和商业化探索，需根据上线后数据决定投入

### 9.3 技术债务与优化空间

| 项目 | 说明 | 计划版本 |
|-----|------|---------|
| 背景动态元素性能优化 | 低端机降低动画帧率或粒子数量 | v1.1.0 |
| 主界面资源预加载 | 进入主界面后异步预加载关卡资源，减少开始挑战后的等待时间 | v1.1.0 |
| 多语言支持 | 按钮文案和提示文字接入i18n系统 | v2.0.0 |
| A/B测试框架 | 对不同按钮文案、颜色、布局进行A/B测试 | v2.1.0 |
| 无障碍适配 | 支持微信小游戏的旁白功能 | v2.2.0 |

### 9.4 风险评估

| 存储空间配额超限 | 低 | 中 | 数据压缩存储，定期清理过期缓存 |
